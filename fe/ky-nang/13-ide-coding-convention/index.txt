18:T15c9,<h1>IDE &amp; Coding Convention</h1>
<p><strong>Định nghĩa.</strong> <strong>IDE</strong> (môi trường lập trình tích hợp, vd VS Code, WebStorm) là công cụ viết code có gợi ý, gỡ lỗi, refactor. <strong>Coding convention</strong> là bộ quy ước viết code thống nhất: đặt tên, format, cấu trúc thư mục, quy tắc commit. Hai thứ này quyết định <strong>tốc độ và sự đồng nhất</strong> của đội: code đọc như một người viết thì review nhanh, bàn giao dễ. Thường tự động hoá bằng <strong>ESLint</strong> (bắt lỗi/chuẩn) + <strong>Prettier</strong> (format); Angular có sẵn <code>ng lint</code>.</p>
<h2>▸ Entry — ② Biết làm</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Dùng tốt IDE (gợi ý, đi tới định nghĩa, refactor đổi tên, gỡ lỗi breakpoint) và <strong>tuân thủ</strong> convention của đội — code nộp đã đúng format/tên, không cần ai sửa tay.</p>
<p><strong>Ví dụ thực tế — dùng IDE &amp; tự áp convention khi lưu.</strong></p>
<pre><code>- F12 / Ctrl+Click: đi tới định nghĩa.  •  F2: đổi tên biến an toàn toàn dự án.
- Breakpoint + Watch: dừng đúng dòng, xem giá trị thay vì rải console.log.
</code></pre>
<pre><code class="language-jsonc"><span class="hljs-comment">// .vscode/settings.json</span>
<span class="hljs-punctuation">{</span> <span class="hljs-attr">&quot;editor.formatOnSave&quot;</span><span class="hljs-punctuation">:</span> <span class="hljs-literal"><span class="hljs-keyword">true</span></span><span class="hljs-punctuation">,</span>
  <span class="hljs-attr">&quot;editor.codeActionsOnSave&quot;</span><span class="hljs-punctuation">:</span> <span class="hljs-punctuation">{</span> <span class="hljs-attr">&quot;source.fixAll.eslint&quot;</span><span class="hljs-punctuation">:</span> <span class="hljs-string">&quot;explicit&quot;</span> <span class="hljs-punctuation">}</span> <span class="hljs-punctuation">}</span>
</code></pre>
<p>Theo đúng quy ước Angular: component <code>PascalCase</code>, file <code>kebab-case.component.ts</code>, biến <code>camelCase</code>.</p>
<p><strong>Vì sao là mức ②:</strong> dùng công cụ thành thạo và tuân thủ chuẩn ổn định.</p>
<h2>▸ Ex·V1 — ② Biết làm (mở rộng phạm vi)</h2>
<p><strong>Khác Entry:</strong> chủ động hơn — cấu hình ESLint/Prettier cho dự án, viết commit message rõ, dùng IDE nâng cao (multi-cursor, snippet, tích hợp Git).</p>
<p><strong>Ví dụ thực tế — &quot;hàng rào&quot; tự động chặn code bẩn vào nhánh chính.</strong></p>
<pre><code class="language-jsonc"><span class="hljs-comment">// .eslintrc — bật quy tắc hay dính</span>
<span class="hljs-punctuation">{</span> <span class="hljs-attr">&quot;rules&quot;</span><span class="hljs-punctuation">:</span> <span class="hljs-punctuation">{</span> <span class="hljs-attr">&quot;@typescript-eslint/no-unused-vars&quot;</span><span class="hljs-punctuation">:</span> <span class="hljs-string">&quot;warn&quot;</span><span class="hljs-punctuation">,</span> <span class="hljs-attr">&quot;eqeqeq&quot;</span><span class="hljs-punctuation">:</span> <span class="hljs-string">&quot;error&quot;</span><span class="hljs-punctuation">,</span>
             <span class="hljs-attr">&quot;@angular-eslint/component-selector&quot;</span><span class="hljs-punctuation">:</span> <span class="hljs-punctuation">[</span><span class="hljs-string">&quot;error&quot;</span><span class="hljs-punctuation">,</span> <span class="hljs-punctuation">{</span> <span class="hljs-attr">&quot;prefix&quot;</span><span class="hljs-punctuation">:</span> <span class="hljs-string">&quot;app&quot;</span> <span class="hljs-punctuation">}</span><span class="hljs-punctuation">]</span> <span class="hljs-punctuation">}</span> <span class="hljs-punctuation">}</span>
</code></pre>

</code></pre>
<p>Commit theo Conventional Commits để lịch sử đọc được:</p>
<pre><code>feat(order): thêm bộ lọc theo trạng thái đơn
fix(auth): sửa token không tự refresh khi hết hạn
</code></pre>
<p><strong>Vì sao vẫn là ②:</strong> thiết lập nề nếp cho dự án, chưa ở mức đặt chuẩn cho nhiều đội.</p>
<h2>▸ Ex·V2 — ③ Thành thạo</h2>
<p><strong>Khác V1:</strong> <strong>đặt và duy trì convention cho cả dự án/đội</strong> — chọn rule có lý do, cân bằng nghiêm vs thực dụng, hướng dẫn người mới.</p>
<p><strong>Ví dụ 1 — chuẩn hoá cấu trúc + rule cho codebase nhiều người.</strong></p>
<pre><code>- Cấu trúc: features/&lt;tên&gt;/ gồm components, services, models — mọi người theo một khuôn.
- Rule: cấm import vòng, cấm 'any' (cảnh báo), bắt trackBy trong *ngFor.
- CI chặn merge nếu lint/format fail → chuẩn được thực thi, không phụ thuộc nhắc nhở.
</code></pre>
<p><strong>Ví dụ 2 — snippet/schematic tăng tốc cả đội.</strong> Bạn tạo Angular schematic hoặc snippet sinh sẵn khung component + test theo đúng convention, để người mới không dựng lệch chuẩn ngay từ đầu.</p>
<p><strong>Vì sao là mức ③:</strong> biến convention thành tài sản chung giúp cả đội đi nhanh và đều.</p>
