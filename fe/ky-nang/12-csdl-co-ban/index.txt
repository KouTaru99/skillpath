18:T1dbc,<h1>Cơ sở dữ liệu cơ bản, web server &amp; microservices</h1>
<p><strong>Định nghĩa.</strong> Ba khái niệm phía sau giao diện mà một FE dev cần hiểu đủ để phối hợp:</p>
<ul>
<li><strong>Cơ sở dữ liệu (CSDL)</strong>: nơi lưu dữ liệu lâu dài. <strong>SQL</strong> (MySQL, PostgreSQL) lưu dạng bảng có quan hệ; <strong>NoSQL</strong> (MongoDB) lưu dạng tài liệu linh hoạt.</li>
<li><strong>Web server / server ứng dụng</strong>: nhận request HTTP và trả response (ví dụ Spring Boot chạy trên JVM).</li>
<li><strong>Microservices</strong>: chia hệ thống thành nhiều dịch vụ nhỏ, mỗi dịch vụ lo một nghiệp vụ.</li>
</ul>
<p>FE không thiết kế CSDL, nhưng hiểu các thứ này giúp biết dữ liệu từ đâu ra, vì sao API có dạng đó. Ví dụ phía server dùng <strong>Java/Spring</strong>.</p>
<h2>▸ Ex·V1 — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Hiểu khác biệt SQL vs NoSQL, biết dữ liệu API phản ánh cấu trúc lưu trữ, hình dung request đi từ trình duyệt tới dịch vụ nào.</p>
<p><strong>Ví dụ thực tế — entity phía server quyết định shape JSON FE nhận.</strong></p>
<pre><code class="language-java"><span class="hljs-meta">@Entity</span>
<span class="hljs-keyword">public</span> <span class="hljs-keyword">class</span> <span class="hljs-title class_">Order</span> {
  <span class="hljs-meta">@Id</span> <span class="hljs-meta">@GeneratedValue</span> Long id;
  String status;
  <span class="hljs-meta">@ManyToOne</span> User customer;         <span class="hljs-comment">// quan hệ → JSON có object customer lồng</span>
  <span class="hljs-meta">@OneToMany(mappedBy = &quot;order&quot;)</span> List&lt;OrderItem&gt; items;   <span class="hljs-comment">// → JSON có mảng items</span>
}
</code></pre>
<p>Bạn đọc entity này là hình dung được API <code>GET /orders/1</code> trả <code>{ id, status, customer:{...}, items:[...] }</code> — nên FE map thẳng ra UI.</p>
<p><strong>Vì sao là mức ①:</strong> hiểu khái niệm đủ để làm việc với dữ liệu, chưa đi sâu hiệu năng.</p>
<h2>▸ Ex·V2 — ① Nhập môn (mở rộng phạm vi)</h2>
<p><strong>Khác V1:</strong> nối được bức tranh <strong>nhiều dịch vụ</strong> — một màn có thể gọi nhiều microservice, và vì sao đôi khi cùng một thứ đến từ hai nguồn.</p>
<p><strong>Ví dụ thực tế — một trang, nhiều dịch vụ.</strong></p>
<pre><code>GET /order-service/orders/123      → thông tin đơn (chỉ lưu userId, sku)
GET /user-service/users/45         → tên, địa chỉ khách
GET /product-service/products?ids=A1,B3 → tên/ảnh sản phẩm
</code></pre>
<p>Mỗi service sở hữu dữ liệu riêng (đơn hàng chỉ lưu <code>userId</code>, <code>sku</code> chứ không lưu sẵn tên) → FE phải gọi thêm để ghép, nên nên gọi song song hoặc nhờ API gateway tổng hợp.</p>
<p><strong>Vì sao vẫn là ①:</strong> ghép được bức tranh hệ thống ở mức hiểu.</p>
<h2>▸ Ex·V3 — ② Biết làm</h2>
<p><strong>Khác V2:</strong> biết <strong>định cỡ sơ bộ hiệu năng</strong> dựa trên cách dữ liệu được lưu/truy vấn, đối thoại có trọng lượng với Back-end.</p>
<p><strong>Ví dụ 1 — phát hiện và đề xuất sửa N+1 ở tầng truy vấn.</strong> Trang hiện 50 đơn, backend lặp 50 lần query lấy customer → chậm.</p>
<pre><code class="language-java"><span class="hljs-comment">// ❌ N+1: mỗi order lại query customer</span>
<span class="hljs-comment">// ✅ join sẵn bằng fetch trong JPQL</span>
<span class="hljs-meta">@Query(&quot;SELECT o FROM Order o JOIN FETCH o.customer WHERE o.status = :st&quot;)</span>
List&lt;Order&gt; <span class="hljs-title function_">findByStatusWithCustomer</span><span class="hljs-params">(String st)</span>;
</code></pre>
<p>Bạn nêu vấn đề bằng ngôn ngữ backend hiểu (N+1, JOIN FETCH, batch).</p>
<p><strong>Ví dụ 2 — vì sao thêm index giúp query nhanh.</strong> Màn tìm đơn theo mã KH chậm dần khi dữ liệu lớn. Bạn hiểu: cột lọc thường xuyên nên có <strong>index</strong> (như mục lục sách) để DB không quét cả bảng — và trao đổi với backend để thêm.</p>
<pre><code class="language-sql"><span class="hljs-keyword">CREATE</span> INDEX idx_orders_customer <span class="hljs-keyword">ON</span> orders(customer_id);
</code></pre>
<p><strong>Vì sao là mức ②:</strong> không chỉ tiêu thụ API mà còn góp phần làm nó hiệu quả.</p>
<h2>▸ Senior·V1 — ③ Thành thạo</h2>
<p><strong>Khác Ex·V3:</strong> không chỉ tối ưu một câu query mà nhận ra khi <strong>ranh giới dữ liệu giữa các service bị phá</strong> — nguyên tắc &quot;database per service&quot; của microservices, và biết đề xuất sửa khi thiết kế đang vi phạm.</p>
<p><strong>Ví dụ — phát hiện anti-pattern &quot;shared database&quot;.</strong></p>
<pre><code>❌ order-service VÀ shipping-service CÙNG đọc thẳng bảng `orders` của nhau
   → đổi schema ở order-service làm sập shipping-service, hai team khoá chặt vào nhau

✅ shipping-service lưu bản sao rút gọn (orderId, address) qua sự kiện
   OrderCreated → shipping-service tự cập nhật bản sao của mình
   → mỗi service tự chủ dữ liệu, đổi schema nội bộ không ảnh hưởng service khác
</code></pre>
<p>Bạn nêu vấn đề này khi review tài liệu thiết kế, trước khi nó thành nợ kỹ thuật khó gỡ.</p>
<p><strong>Vì sao là mức ③:</strong> bạn đọc được thiết kế dữ liệu ở tầm <strong>quan hệ giữa các service</strong>, không chỉ tầm một câu query.</p>
<h2>▸ Senior·V3 — ④ Chuyên sâu</h2>
<p><strong>Khác Senior·V1:</strong> đề xuất được <strong>cách đồng bộ dữ liệu</strong> giữa các service đã tách đúng ranh giới (không chỉ phát hiện ranh giới bị phá).</p>
<p><strong>Ví dụ thực tế — đồng bộ bản sao dữ liệu qua sự kiện (event-driven), thay vì gọi API chờ nhau.</strong> Từ vấn đề đã nêu ở Senior·V1 (<code>shipping-service</code> cần bản sao địa chỉ từ <code>order-service</code>), bạn đề xuất cụ thể cách đồng bộ:</p>
<pre><code>order-service: khi tạo đơn → phát sự kiện OrderCreated { orderId, address } lên message queue
shipping-service: lắng nghe OrderCreated → tự lưu bản sao (orderId, address) vào CSDL riêng
</code></pre>
<p>Khác với việc <code>shipping-service</code> gọi API sang <code>order-service</code> mỗi lần cần địa chỉ (chậm hơn, và nếu <code>order-service</code> đang bảo trì thì <code>shipping-service</code> cũng bị ảnh hưởng theo) — đồng bộ qua sự kiện giúp <code>shipping-service</code> luôn có sẵn dữ liệu cục bộ, đọc nhanh, và <strong>không phụ thuộc runtime</strong> vào service kia còn sống hay không.</p>
<p><strong>Vì sao là mức ④:</strong> bạn không chỉ chỉ ra vấn đề mà <strong>thiết kế được giải pháp</strong> đồng bộ dữ liệu giữa các service, cân nhắc đúng đánh đổi (độ trễ đồng bộ nhỏ, đổi lấy tính độc lập).</p>
