18:T1ec7,<h1>Lập trình fullstack</h1>
<p><strong>Định nghĩa.</strong> Tự viết được cả Front-end lẫn Back-end cho một tính năng trọn vẹn. Khác <a href="/fe/ky-nang/12-csdl-co-ban">CSDL cơ bản + web server</a> (đọc hiểu đủ để phối hợp): ở đây bạn <strong>tự tay code</strong> phần Back-end (thường <strong>Java/Spring</strong> theo stack career-path VCS) khi cần — không chỉ đọc hiểu code người khác viết.</p>
<h2>▸ Senior·V2 — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Tự viết được một API Back-end đơn giản (CRUD) bằng Java/Spring Boot để phục vụ tính năng mình đang làm, khi team Back-end bận hoặc thiếu người — không chỉ đọc hiểu code Back-end.</p>
<p><strong>Ví dụ thực tế — tự viết API &quot;ghi chú nội bộ cho đơn hàng&quot; (BE team chưa kịp làm).</strong></p>
<pre><code class="language-java"><span class="hljs-meta">@RestController</span>
<span class="hljs-meta">@RequestMapping(&quot;/api/orders/{orderId}/notes&quot;)</span>

<p>Bạn tự code đủ để tính năng FE của mình có API dùng ngay, thay vì bị chặn (block) chờ Back-end rảnh tay — miễn tính năng đủ đơn giản (CRUD cơ bản, không đụng nghiệp vụ phức tạp hay bảo mật nhạy cảm).</p>
<p><strong>Vì sao là mức ①:</strong> bạn tự viết được API đơn giản để không bị phụ thuộc hoàn toàn vào Back-end — chưa đảm nhận được phần Back-end phức tạp (giao dịch, bảo mật, hiệu năng) một mình.</p>
<h2>▸ Senior·V3 — ② Biết làm</h2>
<p><strong>Khác V1:</strong> tự xử lý được phần Back-end có <strong>giao dịch (transaction)</strong> — nhiều bước phải cùng thành công hoặc cùng thất bại, không để dữ liệu lệch nửa chừng.</p>
<p><strong>Ví dụ thực tế — API &quot;chuyển đơn hàng sang trạng thái đã thanh toán&quot; phải cập nhật 2 bảng cùng lúc.</strong></p>
<pre><code class="language-java"><span class="hljs-meta">@Transactional</span>   <span class="hljs-comment">// nếu bước 2 lỗi, bước 1 cũng tự động bị huỷ (rollback) — không lệch dữ liệu</span>
<span class="hljs-keyword">public</span> <span class="hljs-keyword">void</span> <span class="hljs-title function_">markAsPaid</span><span class="hljs-params">(Long orderId)</span> {
  <span class="hljs-type">Order</span> <span class="hljs-variable">order</span> <span class="hljs-operator">=</span> orderRepo.findById(orderId).orElseThrow();
  order.setStatus(<span class="hljs-string">&quot;PAID&quot;</span>);
  orderRepo.save(order);                                    <span class="hljs-comment">// bước 1</span>

  inventoryRepo.decreaseStock(order.getProductId(), order.getQty());  <span class="hljs-comment">// bước 2 — trừ kho</span>
}
</code></pre>
<p>Không có <code>@Transactional</code>, nếu bước 2 (trừ kho) lỗi giữa chừng, đơn hàng đã bị đánh dấu &quot;đã thanh toán&quot; nhưng kho chưa trừ — dữ liệu lệch nhau vĩnh viễn. Bạn hiểu và áp dụng đúng ranh giới giao dịch, không chỉ viết CRUD đơn lẻ.</p>
<p><strong>Vì sao là mức ②:</strong> bạn xử lý đúng các thao tác Back-end nhiều bước cần toàn vẹn dữ liệu — không chỉ viết được API CRUD đơn giản từng bảng riêng lẻ.</p>
<h2>▸ Specialist·V2 — ③ Thành thạo</h2>
<p><strong>Khác V1:</strong> tự viết được phần <strong>kiểm quyền (authorization)</strong> đúng cách ở Back-end, không chỉ dựa vào FE ẩn nút — nhắc lại đúng nguyên tắc đã học ở <a href="/fe/ky-nang/03-lap-trinh-an-toan">Lập trình an toàn</a> nhưng lần này tự tay code phần đó.</p>
<p><strong>Ví dụ thực tế — API &quot;xoá đơn hàng&quot; chỉ admin mới được gọi, kiểm tra Ở BACKEND chứ không chỉ ẩn nút FE.</strong></p>
<pre><code class="language-java"><span class="hljs-meta">@DeleteMapping(&quot;/api/orders/{id}&quot;)</span>
<span class="hljs-meta">@PreAuthorize(&quot;hasRole(&#x27;ADMIN&#x27;)&quot;)</span>   <span class="hljs-comment">// Spring Security tự chặn nếu không phải role ADMIN</span>
<span class="hljs-keyword">public</span> ResponseEntity&lt;Void&gt; <span class="hljs-title function_">delete</span><span class="hljs-params">(<span class="hljs-meta">@PathVariable</span> Long id, Authentication auth)</span> {
  <span class="hljs-type">Order</span> <span class="hljs-variable">order</span> <span class="hljs-operator">=</span> orderRepo.findById(id).orElseThrow();
  <span class="hljs-keyword">if</span> (!canManage(auth, order)) {
    <span class="hljs-keyword">throw</span> <span class="hljs-keyword">new</span> <span class="hljs-title class_">AccessDeniedException</span>(<span class="hljs-string">&quot;Không có quyền xoá đơn này&quot;</span>);   <span class="hljs-comment">// kiểm thêm ở mức nghiệp vụ</span>
  }
  orderRepo.delete(order);
  <span class="hljs-keyword">return</span> ResponseEntity.noContent().build();
}
</code></pre>
<p>Bạn tự viết đủ 2 lớp kiểm quyền: <code>@PreAuthorize</code> (kiểm role chung) và kiểm nghiệp vụ cụ thể (<code>canManage</code>, ví dụ chỉ được xoá đơn thuộc chi nhánh mình quản lý) — không dừng ở việc ẩn nút &quot;Xoá&quot; phía FE như mức Senior đã cảnh báo.</p>
<p><strong>Vì sao là mức ③:</strong> bạn tự tay hiện thực được lớp bảo mật Back-end quan trọng nhất (kiểm quyền), không chỉ viết CRUD/transaction — mức cao nhất của kỹ năng fullstack trong thang này.</p>
