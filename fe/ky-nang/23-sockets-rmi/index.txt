18:T1ede,<h1>Lập trình giao tiếp mạng (Sockets, RMI)</h1>
<p><strong>Định nghĩa.</strong> <strong>Socket</strong> là giao tiếp mạng cấp thấp qua TCP/UDP — hai máy mở một <strong>kết nối</strong> rồi gửi/nhận dữ liệu qua lại, khác hẳn HTTP thông thường (mỗi request tự mở rồi đóng kết nối ngay). <strong>RMI</strong> (Remote Method Invocation, phổ biến ở Java) cho phép gọi hàm trên máy khác gần như gọi hàm cục bộ — khái niệm ít khi FE chạm trực tiếp, nhưng cần hiểu để đối thoại với Back-end Java. Với Front-end, hiện thân thật của &quot;lập trình socket&quot; chính là <strong>WebSocket</strong>: kết nối luôn mở, hai chiều, dùng cho real-time (chat, thông báo trực tiếp) — khác HTTP là bạn không phải hỏi lại &quot;có gì mới không&quot;, server tự đẩy dữ liệu khi có.</p>
<h2>▸ Senior·V2 — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Hiểu khác biệt HTTP (request–response, đóng kết nối ngay) và WebSocket (kết nối luôn mở, hai chiều), dùng WebSocket API cơ bản để nhận cập nhật real-time.</p>
<p><strong>Ví dụ thực tế — thông báo đơn hàng đổi trạng thái, đẩy trực tiếp không cần tải lại (Angular).</strong></p>
<pre><code class="language-typescript"><span class="hljs-meta">@Injectable</span>({ <span class="hljs-attr">providedIn</span>: <span class="hljs-string">&#x27;root&#x27;</span> })

<p>Nếu dùng HTTP thông thường, muốn &quot;real-time&quot; phải <strong>poll</strong> (gọi API lặp lại mỗi vài giây để hỏi &quot;có gì mới chưa&quot;) — tốn tài nguyên và vẫn trễ vài giây. WebSocket giữ kết nối mở, server đẩy tin ngay khi có, không cần hỏi.</p>
<p><strong>Vì sao là mức ①:</strong> bạn dùng được kết nối real-time cho trường hợp đơn giản (một luồng cập nhật) — chưa xử lý các vấn đề khó hơn (mất kết nối, reconnect, nhiều kênh cùng lúc).</p>
<h2>▸ Senior·V3 — ② Biết làm</h2>
<p><strong>Khác V1:</strong> xử lý được <strong>mất kết nối và tự kết nối lại</strong> — vấn đề thật của WebSocket mà ví dụ cơ bản ở V1 chưa chạm tới.</p>
<p><strong>Ví dụ thực tế — tự động reconnect khi mất mạng, không để người dùng phải tự tải lại trang.</strong></p>

<p>Không có cơ chế này, người dùng ngồi trong quán cà phê wifi chập chờn sẽ mất thông báo real-time mà không hề biết, tưởng ứng dụng vẫn hoạt động bình thường.</p>
<p><strong>Vì sao là mức ②:</strong> bạn xử lý được vấn đề thật của kết nối real-time trong môi trường mạng không ổn định — không chỉ code cho trường hợp mạng luôn tốt.</p>
<h2>▸ Specialist·V2 — ③ Thành thạo</h2>
<p><strong>Khác V1:</strong> hiểu vấn đề khi hệ thống có <strong>nhiều server cùng xử lý WebSocket</strong> — một vấn đề chỉ lộ ra khi hệ thống có tải cao, cần scale nhiều instance.</p>
<p><strong>Ví dụ thực tế — thông báo không tới khi có 2 server, vì client và người gửi không cùng kết nối vào 1 server.</strong> Hệ thống chạy 2 instance API để chịu tải cao hơn. Client A kết nối WebSocket vào server 1. Khi có sự kiện cần báo cho client A, nhưng code xử lý sự kiện đó lại chạy trên server 2 — server 2 không biết client A đang giữ kết nối ở server 1, nên không gửi được gì. Bạn nhận diện đây là vấn đề cần một <strong>backplane</strong> (kênh trung gian, ví dụ Redis Pub/Sub) để mọi server đều biết và chuyển tiếp được tin nhắn cho đúng client, bất kể client đang kết nối vào server nào.</p>
<p><strong>Vì sao là mức ③:</strong> bạn hiểu được giới hạn của WebSocket khi hệ thống scale ra nhiều server — vấn đề chỉ xuất hiện ở quy mô thật, không thấy được khi chỉ chạy 1 server lúc code/test.</p>
