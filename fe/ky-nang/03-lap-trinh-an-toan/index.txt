18:T25fb,<h1>Lập trình an toàn (Secure Coding)</h1>
<p><strong>Định nghĩa.</strong> Viết code phòng ngừa lỗ hổng bảo mật ngay từ khâu lập trình. Tham chiếu phổ biến là <strong>OWASP Top 10</strong> (injection, XSS, hỏng xác thực/phân quyền, lộ dữ liệu nhạy cảm...). Ở phía Front-end với Angular, trọng tâm: chống <strong>XSS</strong>, xử lý <strong>token</strong> an toàn, dùng đúng cơ chế <strong>CSRF/sanitization</strong> Angular cung cấp, và không lộ <strong>secret</strong>. Với công ty an ninh mạng, đây là kỹ năng bắt buộc xuyên suốt mọi level.</p>
<h2>▸ Entry — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Biết các lỗ hổng phổ biến, không hardcode secret trong code FE, và hiểu Angular <strong>tự escape</strong> dữ liệu trong template — đừng phá cơ chế đó.</p>
<p><strong>Ví dụ thực tế — Angular tự chống XSS, chỉ hỏng khi bạn ép bỏ qua.</strong></p>
<pre><code class="language-typescript"><span class="hljs-comment">// ✅ AN TOÀN: interpolation tự escape, &lt;script&gt; của kẻ xấu thành text vô hại</span>
<span class="hljs-meta">@Component</span>({ <span class="hljs-attr">template</span>: <span class="hljs-string">`&lt;div&gt;{{ user.bio }}&lt;/div&gt;`</span> })

<span class="hljs-comment">// ❌ NGUY HIỂM: bypassSecurityTrustHtml tắt lá chắn → XSS trở lại</span>
<span class="hljs-variable language_">this</span>.<span class="hljs-property">safeBio</span> = <span class="hljs-variable language_">this</span>.<span class="hljs-property">sanitizer</span>.<span class="hljs-title function_">bypassSecurityTrustHtml</span>(user.<span class="hljs-property">bio</span>);
</code></pre>
<p>Quy tắc nhập môn: mặc định để Angular escape; chỉ đụng <code>bypassSecurityTrust*</code> khi thật hiểu và dữ liệu đã sạch.</p>
<p><strong>Vì sao là mức ①:</strong> nhận ra rủi ro và tránh lỗi cơ bản, chưa áp dụng nhất quán toàn luồng.</p>
<h2>▸ Ex·V1 — ② Biết làm</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Áp dụng nhất quán — dùng <code>DomSanitizer.sanitize()</code> khi buộc render HTML người dùng, hiểu đánh đổi nơi lưu token, không đưa dữ liệu nhạy cảm lên URL.</p>
<p><strong>Ví dụ thực tế — render rich text an toàn + lưu token.</strong></p>
<pre><code class="language-typescript"><span class="hljs-comment">// Làm sạch HTML trước khi bind [innerHTML]</span>
<span class="hljs-keyword">const</span> clean = <span class="hljs-variable language_">this</span>.<span class="hljs-property">sanitizer</span>.<span class="hljs-title function_">sanitize</span>(<span class="hljs-title class_">SecurityContext</span>.<span class="hljs-property">HTML</span>, richHtml);
</code></pre>
<p>Đánh đổi nơi lưu token:</p>
<pre><code>localStorage : tiện, nhưng JS đọc được → dính XSS là mất token.
httpOnly cookie: JS không đọc được (an toàn hơn XSS), nhưng phải chống CSRF.
</code></pre>
<pre><code class="language-typescript"><span class="hljs-comment">// ❌ token trên URL → bị log ở server/proxy/lịch sử</span>
<span class="hljs-variable language_">this</span>.<span class="hljs-property">http</span>.<span class="hljs-title function_">get</span>(<span class="hljs-string">`/api/profile?token=<span class="hljs-subst">${token}</span>`</span>);
<span class="hljs-comment">// ✅ token trong header</span>
<span class="hljs-variable language_">this</span>.<span class="hljs-property">http</span>.<span class="hljs-title function_">get</span>(<span class="hljs-string">&#x27;/api/profile&#x27;</span>, { <span class="hljs-attr">headers</span>: { <span class="hljs-title class_">Authorization</span>: <span class="hljs-string">`Bearer <span class="hljs-subst">${token}</span>`</span> } });
</code></pre>
<p><strong>Vì sao là mức ②:</strong> bảo mật thành thói quen trong từng task.</p>
<h2>▸ Ex·V2 — ② Biết làm (mở rộng phạm vi)</h2>
<p><strong>Khác V1:</strong> rà soát <strong>toàn luồng</strong> — bật cơ chế CSRF của Angular, hiểu CSP cơ bản, không log dữ liệu nhạy cảm.</p>
<p><strong>Ví dụ 1 — bật chống CSRF sẵn có của Angular.</strong> Angular <code>HttpClientXsrfModule</code> tự đọc cookie XSRF-TOKEN và gắn header cho request không phải GET:</p>
<pre><code class="language-typescript"><span class="hljs-attr">imports</span>: [ <span class="hljs-title class_">HttpClientXsrfModule</span>.<span class="hljs-title function_">withOptions</span>({ <span class="hljs-attr">cookieName</span>: <span class="hljs-string">&#x27;XSRF-TOKEN&#x27;</span>, <span class="hljs-attr">headerName</span>: <span class="hljs-string">&#x27;X-XSRF-TOKEN&#x27;</span> }) ]
</code></pre>
<p>Backend chỉ cần set cookie và kiểm header khớp — chặn request giả từ site khác.</p>
<p><strong>Ví dụ 2 — không rò dữ liệu qua log.</strong> Interceptor log request để debug nhưng lỡ in cả body chứa mật khẩu. Bạn lọc field nhạy cảm trước khi log, và tắt log chi tiết ở production.</p>
<p><strong>Vì sao vẫn là ②:</strong> rà soát tốt toàn luồng, chưa ở mức thiết kế phòng thủ / review cho team.</p>
<h2>▸ Ex·V3 — ③ Thành thạo</h2>
<p><strong>Khác V2:</strong> chủ động <strong>đề xuất giải pháp bảo mật</strong>, thiết kế chống tấn công, <strong>review code bắt lỗ hổng</strong> cho người khác.</p>
<p><strong>Ví dụ 1 — bắt open redirect khi review.</strong></p>
<pre><code class="language-typescript"><span class="hljs-comment">// ❌ chuyển hướng theo tham số người dùng → dụ tới site lừa đảo</span>
<span class="hljs-variable language_">window</span>.<span class="hljs-property">location</span>.<span class="hljs-property">href</span> = params[<span class="hljs-string">&#x27;returnUrl&#x27;</span>];
<span class="hljs-comment">// ✅ whitelist đường dẫn nội bộ</span>
<span class="hljs-keyword">const</span> allowed = [<span class="hljs-string">&#x27;/dashboard&#x27;</span>, <span class="hljs-string">&#x27;/profile&#x27;</span>, <span class="hljs-string">&#x27;/orders&#x27;</span>];
<span class="hljs-variable language_">this</span>.<span class="hljs-property">router</span>.<span class="hljs-title function_">navigateByUrl</span>(allowed.<span class="hljs-title function_">includes</span>(url) ? url : <span class="hljs-string">&#x27;/dashboard&#x27;</span>);
</code></pre>
<p><strong>Ví dụ 2 — chặn truy cập route bằng guard + hiểu giới hạn của nó.</strong></p>
<pre><code class="language-typescript"><span class="hljs-meta">@Injectable</span>({ <span class="hljs-attr">providedIn</span>: <span class="hljs-string">&#x27;root&#x27;</span> })
<span class="hljs-keyword">export</span> <span class="hljs-keyword">class</span> <span class="hljs-title class_">AdminGuard</span> <span class="hljs-keyword">implements</span> <span class="hljs-title class_">CanActivate</span> {
  <span class="hljs-title function_">canActivate</span>(): <span class="hljs-built_in">boolean</span> { <span class="hljs-keyword">return</span> <span class="hljs-variable language_">this</span>.<span class="hljs-property">auth</span>.<span class="hljs-title function_">hasRole</span>(<span class="hljs-string">&#x27;admin&#x27;</span>); }
}
</code></pre>
<p>Bạn nhấn với team: guard chỉ chặn UI — <strong>API vẫn phải tự kiểm quyền</strong>, vì kẻ xấu gọi thẳng API không qua router.</p>
<p><strong>Ví dụ 3 — thiết lập Content Security Policy.</strong> Đề xuất header CSP để trình duyệt chỉ chạy script từ nguồn tin cậy, chặn inline script lạ — một lớp phòng thủ nữa cho XSS, ghi vào checklist trước khi lên production.</p>
<p><strong>Vì sao là mức ③:</strong> bảo vệ cả sản phẩm chứ không chỉ code của mình, nâng chuẩn an toàn cho đội.</p>
<h2>▸ Senior·V3 — ④ Chuyên sâu</h2>
<p><strong>Khác Ex·V3:</strong> không chỉ review bắt lỗ hổng đã có mà <strong>đánh giá rủi ro trước khi code</strong> (threat modeling) cho tính năng mới, ngay từ khâu thiết kế.</p>
<p><strong>Ví dụ thực tế — threat modeling nhanh cho tính năng &quot;chia sẻ đơn hàng qua link công khai&quot;.</strong> Trước khi ai viết dòng code nào, bạn đặt câu hỏi phòng thủ:</p>
<pre><code>- Link công khai có đoán được không? (nếu dùng orderId=123 tuần tự → kẻ xấu dò hết đơn của người khác)
  → đổi sang token ngẫu nhiên dài, không đoán được.
- Link có hết hạn không? (nếu không, lộ vĩnh viễn dù đơn đã xong từ lâu)
  → đặt thời hạn hết hạn (vd 7 ngày).
- Ai xem được link có thấy thông tin nhạy cảm không? (địa chỉ, số điện thoại đầy đủ)
  → ẩn bớt phần nhạy cảm ở bản xem công khai.
</code></pre>
<p>Ba câu hỏi này được đặt ra <strong>trước khi code</strong>, khi sửa vẫn còn rẻ — nếu để tới lúc review mới phát hiện <code>orderId</code> tuần tự bị lộ, sửa đã tốn công hơn nhiều (đổi cả schema, migrate dữ liệu cũ).</p>
<p><strong>Vì sao là mức ④:</strong> bạn chủ động đánh giá rủi ro bảo mật ở khâu thiết kế, trước khi có code để review — mức phòng thủ sâu nhất, rẻ nhất.</p>
