18:T41b9,<h1>Microfrontends</h1>
<p><strong>Định nghĩa.</strong> <strong>Microfrontends (MFE)</strong> là kiến trúc chia một ứng dụng Front-end lớn thành nhiều mảnh <strong>build &amp; deploy độc lập</strong>, rồi <strong>ghép lại lúc chạy (runtime)</strong> thành một trang người dùng thấy liền mạch. Mỗi mảnh do một đội sở hữu, <strong>có thể khác framework/version</strong> (Angular 12 · React · Angular 15…). Điều làm nên microfrontend <strong>không phải</strong> số lượng repo, mà là: <em>mỗi mảnh deploy độc lập + ghép lúc runtime</em>.</p>
<p><strong>Đừng lẫn với monorepo.</strong> Đây là hai trục <strong>vuông góc</strong>, không phải một:</p>
<ul>
<li><strong>Monorepo / poly-repo</strong> = câu hỏi <em>&quot;code nằm ở đâu&quot;</em> (một repo nhiều app, hay mỗi app một repo). Chuyện <strong>build-time / tổ chức mã</strong>.</li>
<li><strong>Microfrontend</strong> = câu hỏi <em>&quot;các mảnh deploy &amp; ghép thế nào&quot;</em>. Chuyện <strong>runtime / kiến trúc</strong>.</li>
</ul>
<p>Bạn có thể monorepo mà mỗi app vẫn deploy riêng (là microfrontend), hoặc poly-repo (cũng microfrontend). Nên <strong>đừng đo microfrontend bằng số repo</strong>.</p>
<pre><code>┌────────────────────────────────────────────────────┐
│  Trình duyệt — người dùng thấy 1 trang liền mạch     │
│  ┌──────────────────────────────────────────────┐   │
│  │  SHELL (host) — router nạp MFE theo route       │  │
│  └─────┬─────────────┬─────────────┬────────────┘   │
│        │             │             │                │
│  ┌─────▼─────┐ ┌─────▼─────┐ ┌─────▼──────┐         │
│  │ Sản phẩm  │ │ Giỏ hàng  │ │ Tài khoản  │         │
│  │ Angular12 │ │ React     │ │ Angular15  │         │
│  │ Team A    │ │ Team B    │ │ Team C     │         │
│  └─────┬─────┘ └─────┬─────┘ └─────┬──────┘         │
│        └─────────────┴─────────────┘                │
│              Event bus (chia sẻ state)              │
└────────────────────────────────────────────────────┘
 Mỗi MFE: repo riêng · build riêng · deploy riêng · URL riêng
</code></pre>
<p><strong>Các kiểu ghép (khác nhau ở &quot;đa framework/version được tới đâu&quot;):</strong></p>
<table>
<thead>
<tr>
<th>Kiểu tích hợp</th>
<th>Đa framework/version?</th>
<th>Đánh đổi</th>
</tr>
</thead>
<tbody>
<tr>
<td>Web Components / iframe / <strong>single-spa</strong> (cô lập runtime)</td>
<td>✅ thoải mái</td>
<td>mỗi mảnh tự mang runtime → tải trùng, nặng hơn</td>
</tr>
<tr>
<td><strong>Webpack Module Federation</strong> + <code>shared: singleton</code></td>
<td>⚠️ buộc <strong>cùng/tương thích version</strong> framework</td>
<td>nhẹ (dùng chung 1 bản), nhưng lệ thuộc version</td>
</tr>
<tr>
<td>Module Federation <strong>không</strong> share framework</td>
<td>✅ đa version</td>
<td>mỗi mảnh tự bundle → tải trùng</td>
</tr>
</tbody>
</table>
<p>Và <strong>router không phải kênh giao tiếp duy nhất</strong>: router lo <em>điều hướng</em> (ghép mảnh theo URL); còn chia sẻ <em>state/dữ liệu</em> (đăng nhập, giỏ hàng) thì cần thêm <strong>event bus / custom event / shared state</strong>.</p>
<h2>▸ Ex·V2 — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Hiểu microfrontend là gì, <strong>phân biệt với monorepo</strong>, biết các kiểu ghép, và một mảnh phải <strong>tự chứa</strong> (không phụ thuộc ngầm mảnh khác).</p>
<p><strong>Cấu trúc thư mục — hai cách để cùng ra một kiến trúc microfrontend:</strong></p>
<pre><code># Cách A — poly-repo (mỗi MFE một repo, độc lập nhất)
shell-app/         → repo Host: router trỏ tới từng MFE
products-mfe/      → repo Team A (Angular 12), tự CI/CD lên products.cty.com
cart-mfe/          → repo Team B (React)
account-mfe/       → repo Team C (Angular 15)

# Cách B — monorepo (Nx) — VẪN là microfrontend
my-workspace/
  apps/  shell/  products/  cart/  account/   ← mỗi app vẫn build &amp; deploy RIÊNG
  libs/  shared-ui/  shared-auth/             ← code dùng chung (lợi thế monorepo)
</code></pre>
<p><strong>Ví dụ ghép cơ bản (Angular 12+ / Webpack 5, dùng <code>@angular-architects/module-federation</code>).</strong> Mảnh &quot;products&quot; phơi module ra:</p>
<pre><code class="language-javascript"><span class="hljs-comment">// products-mfe/webpack.config.js</span>
<span class="hljs-keyword">new</span> <span class="hljs-title class_">ModuleFederationPlugin</span>({
  <span class="hljs-attr">name</span>: <span class="hljs-string">&#x27;products&#x27;</span>,
  <span class="hljs-attr">filename</span>: <span class="hljs-string">&#x27;remoteEntry.js&#x27;</span>,                 <span class="hljs-comment">// shell sẽ tải file này</span>
  <span class="hljs-attr">exposes</span>: { <span class="hljs-string">&#x27;./Module&#x27;</span>: <span class="hljs-string">&#x27;./src/app/products/products.module.ts&#x27;</span> },
});
</code></pre>
<p>Shell nạp mảnh đó theo route — <strong>điểm mấu chốt là URL <code>remoteEntry</code></strong> (nơi mảnh đang deploy):</p>
<pre><code class="language-typescript">{
  <span class="hljs-attr">path</span>: <span class="hljs-string">&#x27;products&#x27;</span>,
  <span class="hljs-attr">loadChildren</span>: <span class="hljs-function">() =&gt;</span> <span class="hljs-title function_">loadRemoteModule</span>({
    <span class="hljs-attr">type</span>: <span class="hljs-string">&#x27;module&#x27;</span>,
    <span class="hljs-attr">remoteEntry</span>: <span class="hljs-string">&#x27;https://products.cty.com/remoteEntry.js&#x27;</span>,   <span class="hljs-comment">// ← mảnh &quot;ở&quot; đâu</span>
    <span class="hljs-attr">exposedModule</span>: <span class="hljs-string">&#x27;./Module&#x27;</span>,
  }).<span class="hljs-title function_">then</span>(<span class="hljs-function"><span class="hljs-params">m</span> =&gt;</span> m.<span class="hljs-property">ProductsModule</span>),
}
</code></pre>
<p><strong>Vì sao là mức ①:</strong> nắm khái niệm và ghép được theo mẫu, chưa xử vấn đề khó (chia sẻ dependency, đa framework, giao tiếp).</p>
<h2>▸ Ex·V3 — ② Biết làm</h2>
<p><strong>Khác V2:</strong> <strong>chọn đúng kiểu tích hợp</strong> theo nhu cầu và xử các bài toán thực — chia sẻ dependency (kèm đánh đổi version), giao tiếp giữa mảnh, giữ nhất quán đăng nhập/giao diện.</p>
<p><strong>Ví dụ 1 — cùng stack Angular: Module Federation share singleton (nhẹ, nhưng ràng version).</strong></p>
<pre><code class="language-javascript"><span class="hljs-attr">shared</span>: {
  <span class="hljs-string">&#x27;@angular/core&#x27;</span>:   { <span class="hljs-attr">singleton</span>: <span class="hljs-literal">true</span>, <span class="hljs-attr">strictVersion</span>: <span class="hljs-literal">true</span>, <span class="hljs-attr">requiredVersion</span>: <span class="hljs-string">&#x27;^12.0.0&#x27;</span> },
  <span class="hljs-string">&#x27;@angular/common&#x27;</span>: { <span class="hljs-attr">singleton</span>: <span class="hljs-literal">true</span>, <span class="hljs-attr">strictVersion</span>: <span class="hljs-literal">true</span>, <span class="hljs-attr">requiredVersion</span>: <span class="hljs-string">&#x27;^12.0.0&#x27;</span> },
  <span class="hljs-attr">rxjs</span>:              { <span class="hljs-attr">singleton</span>: <span class="hljs-literal">true</span> },
}
</code></pre>
<p><code>singleton: true</code> để shell và các mảnh <strong>dùng chung một bản Angular</strong> (đỡ tải trùng, tránh hai bản Angular đập nhau). <strong>Đánh đổi:</strong> các mảnh phải cùng/tương thích major version — đây chính là lý do &quot;MF share singleton&quot; <strong>không</strong> cho đa version tự do.</p>
<p><strong>Ví dụ 2 — cần đa framework/version thật (Angular + React): đừng share, hãy cô lập.</strong>
Khi các mảnh khác framework, bỏ share framework (mỗi mảnh tự bundle), hoặc gói mảnh thành <strong>Web Component</strong> để shell nhúng như thẻ HTML thường:</p>
<pre><code class="language-html"><span class="hljs-comment">&lt;!-- shell nhúng mảnh React như custom element, không cần biết nó viết bằng gì --&gt;</span>
<span class="hljs-tag">&lt;<span class="hljs-name">cart-widget</span> <span class="hljs-attr">user-id</span>=<span class="hljs-string">&quot;42&quot;</span>&gt;</span><span class="hljs-tag">&lt;/<span class="hljs-name">cart-widget</span>&gt;</span>
</code></pre>
<p>Trả giá: tải trùng runtime; đổi lại các mảnh thực sự độc lập công nghệ (đúng lời hứa gốc của microfrontend). Công cụ phổ biến cho hướng này: <strong>single-spa</strong>.</p>
<p><strong>Ví dụ 3 — giao tiếp giữa mảnh qua event bus (không gọi thẳng vào nhau).</strong></p>
<pre><code class="language-typescript"><span class="hljs-comment">// mảnh &quot;cart&quot; phát sự kiện khi thêm hàng</span>
<span class="hljs-variable language_">window</span>.<span class="hljs-title function_">dispatchEvent</span>(<span class="hljs-keyword">new</span> <span class="hljs-title class_">CustomEvent</span>(<span class="hljs-string">&#x27;cart:add&#x27;</span>, { <span class="hljs-attr">detail</span>: { id, qty } }));

</code></pre>
<p>Router điều hướng giữa các mảnh; event bus lo chia sẻ state (giỏ hàng, phiên đăng nhập). Gọi thẳng service của mảnh khác sẽ khoá chặt chúng lại — mất tính độc lập.</p>
<p><strong>Vì sao là mức ②:</strong> chọn được kiểu tích hợp hợp bối cảnh và giải quyết các đánh đổi cốt lõi (version, đa framework, giao tiếp) — không đóng khung microfrontend vào một cách duy nhất.</p>
<h2>▸ Senior·V1 — ③ Thành thạo</h2>
<p><strong>Khác Ex·V3:</strong> không chỉ ghép được mảnh mà <strong>vận hành nó ở production</strong> — CI/CD độc lập cho từng mảnh, kiểm tra hợp đồng giữa shell/remote trước khi deploy, và biết <strong>khi nào KHÔNG nên</strong> chọn microfrontend.</p>
<p><strong>Ví dụ 1 — pipeline deploy độc lập cho một mảnh (không kéo theo cả hệ thống).</strong></p>
<pre><code class="language-yaml"><span class="hljs-comment"># products-mfe/.github/workflows/deploy.yml — chỉ deploy khi mảnh này đổi</span>
<span class="hljs-attr">on:</span>

    <span class="hljs-attr">paths:</span> [<span class="hljs-string">&#x27;products-mfe/**&#x27;</span>]

      <span class="hljs-bullet">-</span> <span class="hljs-attr">run:</span> <span class="hljs-string">npm</span> <span class="hljs-string">run</span> <span class="hljs-string">build</span>
      <span class="hljs-bullet">-</span> <span class="hljs-attr">run:</span> <span class="hljs-string">aws</span> <span class="hljs-string">s3</span> <span class="hljs-string">sync</span> <span class="hljs-string">dist/</span> <span class="hljs-string">s3://products.cty.com</span>   <span class="hljs-comment"># remoteEntry.js mới, shell tự nạp bản mới</span>
</code></pre>
<p>Đây là lợi ích cốt lõi mà PDF career-path gọi là &quot;thành thạo lập trình theo mô hình microfrontends&quot;: mảnh &quot;products&quot; đổi và lên production mà không cần build lại &quot;cart&quot; hay &quot;account&quot;.</p>

<p><strong>Ví dụ 3 — biết khi nào KHÔNG nên dùng microfrontend.</strong> Team 5 người, 1 sản phẩm, không có nhu cầu deploy độc lập theo đội → microfrontend chỉ thêm độ phức tạp vận hành (nhiều pipeline, nhiều nơi theo dõi lỗi, đồng bộ version framework) mà không đổi lại lợi ích gì — ở quy mô này, một Angular app thường (monolith FE) là lựa chọn đúng, không phải &quot;lỗi thời&quot;.</p>
<p><strong>Vì sao là mức ③:</strong> bạn làm chủ microfrontend <strong>ở vòng đời thật</strong> (deploy, kiểm hợp đồng, vận hành) chứ không chỉ ghép được lúc code, và đủ tỉnh táo để không áp nó vào nơi không cần.</p>
<h2>▸ Senior·V2 — ④ Chuyên sâu</h2>
<p><strong>Khác Senior·V1:</strong> không chỉ vận hành một hệ microfrontend có sẵn mà <strong>thiết kế bộ khung (platform) cho các đội khác dựng mảnh mới lên đó</strong> — chuẩn hoá cách một mảnh &quot;gia nhập&quot; hệ thống.</p>
<p><strong>Ví dụ thực tế — bộ khung &quot;onboarding&quot; cho một mảnh mới.</strong> Khi công ty có thêm đội mới muốn thêm mảnh &quot;Khuyến mãi&quot; vào shell hiện có, bạn không để họ tự mò cách tích hợp — bạn định nghĩa sẵn: template repo có cấu hình Module Federation chuẩn, checklist bắt buộc (đăng ký route với shell, đúng version Angular singleton, có healthcheck, có contract test với shell), và tài liệu &quot;thêm một microfrontend mới trong 1 ngày&quot; thay vì vài tuần dò dẫm.</p>
<p><strong>Vì sao là mức ④:</strong> bạn thiết kế được <strong>nền tảng</strong> cho microfrontend mở rộng bền vững qua nhiều đội, không chỉ vận hành tốt hệ hiện có.</p>
