18:T1986,<h1>Đọc hiểu &amp; soi lỗi tài liệu giải pháp hệ thống</h1>
<p><strong>Định nghĩa.</strong> <em>Tài liệu giải pháp hệ thống</em> (solution / design document) mô tả cách hệ thống được dựng: kiến trúc tổng thể, luồng nghiệp vụ (sequence/flow), đặc tả API, mô hình dữ liệu, ràng buộc phi chức năng. Kỹ năng gồm hai vế: (1) <strong>đọc hiểu</strong> để code đúng ý người thiết kế, và (2) <strong>soi lỗi</strong> — phát hiện chỗ thiếu, mâu thuẫn, logic sai <em>trước khi</em> code, vì sửa trên tài liệu rẻ hơn sửa trên code gấp nhiều lần.</p>
<h2>▸ Entry — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Đọc tài liệu nắm phần việc của mình: màn nào, gọi API nào, tham số gì, hiển thị ra sao; biết hỏi lại khi chưa rõ thay vì đoán.</p>
<p><strong>Ví dụ thực tế.</strong> Từ sequence diagram đăng nhập + đặc tả API, bạn xác định FE phải làm gì:</p>
<pre><code>User → FE → POST /api/login {username, password}
       ← 200 { token, expiresIn }  → FE lưu token, chuyển trang chủ
       ← 401 { message }           → FE hiện lỗi &quot;Sai tài khoản/mật khẩu&quot;
</code></pre>
<p>Bạn đọc ra: cần form, gọi đúng endpoint, xử lý 2 nhánh 200/401, lưu token cho request sau.</p>
<p><strong>Vì sao là mức ①:</strong> hiểu đúng để làm đúng, chưa đối chiếu sâu hay phát hiện thiếu sót.</p>
<h2>▸ Ex·V1 — ② Biết làm</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Đọc độc lập, <strong>đối chiếu tài liệu với thực tế</strong> — phát hiện chỗ chưa mô tả (thiếu trạng thái lỗi, thiếu validation, thiếu loading).</p>
<p><strong>Ví dụ thực tế — bắt lỗ hổng trong API spec.</strong></p>
<pre><code class="language-yaml"><span class="hljs-string">GET</span> <span class="hljs-string">/api/users/{id}:</span>
  <span class="hljs-attr">responses:</span>
    <span class="hljs-attr">200:</span> { <span class="hljs-string">id</span>, <span class="hljs-string">name</span>, <span class="hljs-string">email</span> }
    <span class="hljs-comment"># ❓ Thiếu: 404 khi id không tồn tại?</span>
    <span class="hljs-comment"># ❓ Thiếu: 401 khi token hết hạn?</span>
</code></pre>
<p>Bạn nêu ngay: <em>&quot;Spec chưa có 404 và 401 — FE cần biết để hiển thị 'không tìm thấy' và tự đăng xuất khi token hết hạn.&quot;</em> — tiết kiệm cả buổi debug về sau.</p>
<p><strong>Vì sao là mức ②:</strong> soi được chỗ thiếu trong phạm vi một API/màn.</p>
<h2>▸ Ex·V2 — ② Biết làm (mở rộng phạm vi)</h2>
<p><strong>Khác V1:</strong> soi lỗi logic <strong>xuyên màn / xuyên luồng</strong> — mâu thuẫn giữa các phần, thứ tự phụ thuộc sai, giả định không đứng vững.</p>
<p><strong>Ví dụ 1 — mâu thuẫn thứ tự dữ liệu.</strong></p>
<pre><code>Luồng A &quot;Xem giỏ hàng&quot;: giả định mỗi sản phẩm đã có discountPrice.
Luồng B &quot;Áp mã giảm giá&quot;: chỉ tính discountPrice SAU khi nhập mã.
→ A hiển thị discountPrice khi nó chưa tồn tại (B chạy sau) → sai thứ tự.
</code></pre>
<p><strong>Ví dụ 2 — thiếu trạng thái trung gian.</strong> Tài liệu mô tả &quot;bấm Thanh toán → sang trang kết quả&quot;, nhưng không nói lúc chờ cổng thanh toán phản hồi thì hiển thị gì. Bạn chỉ ra và đề xuất trạng thái &quot;đang xử lý&quot; + khoá nút chống bấm 2 lần.</p>
<p><strong>Vì sao vẫn là ②:</strong> bắt lỗi logic liên thông, chưa phản biện cả giải pháp kiến trúc.</p>
<h2>▸ Ex·V3 — ③ Thành thạo</h2>
<p><strong>Khác V2:</strong> <strong>phản biện giải pháp</strong> và đề xuất thiết kế tốt hơn, đánh giá tác động phi chức năng (hiệu năng, bảo mật, mở rộng).</p>
<p><strong>Ví dụ 1 — phản biện gọi API tuần tự.</strong> Dashboard gọi 5 API tuần tự, mỗi cái chờ cái trước (~2.5s). Bạn đề xuất gọi <strong>song song</strong> (<code>forkJoin</code> trong Angular) đưa về ~0.6s, hoặc gộp thành một endpoint tổng; cân nhắc đánh đổi (gộp khó tái dùng hơn).</p>
<p><strong>Ví dụ 2 — chỉ ra rủi ro bảo mật trong thiết kế.</strong> Tài liệu để FE nhận <code>role</code> từ response rồi tự ẩn/hiện nút &quot;Xoá&quot;. Bạn phản biện: ẩn nút chỉ là UX, <strong>backend vẫn phải kiểm quyền</strong> — nếu không, ai gọi thẳng API vẫn xoá được. Đề xuất ghi rõ backend enforce quyền.</p>
<p><strong>Ví dụ 3 — soi rủi ro hiệu năng từ data model.</strong> Thiết kế trả về danh sách đơn kèm toàn bộ chi tiết từng đơn. Bạn chỉ ra với 1.000 đơn payload sẽ khổng lồ → đề xuất trả tóm tắt + lazy-load chi tiết khi mở.</p>
<p><strong>Vì sao là mức ③:</strong> không chỉ bắt lỗi mà còn nâng chất lượng chính bản thiết kế.</p>
<h2>▸ Senior·V2 — ④ Chuyên sâu</h2>
<p><strong>Khác Ex·V3:</strong> soi lỗi ở tầm <strong>liên hệ thống</strong> — mâu thuẫn giữa các service/team khác nhau, không chỉ trong một ứng dụng.</p>
<p><strong>Ví dụ thực tế — hai hệ thống mâu thuẫn nhau về &quot;ai là nguồn chân lý&quot;.</strong> Tài liệu module Đơn hàng ghi: giá sản phẩm lấy trực tiếp từ <code>order.price</code> (lưu tại thời điểm đặt). Tài liệu module Khuyến mãi (team khác) lại giả định giá hiển thị luôn là <code>product.currentPrice</code> mới nhất. Hai tài liệu không xung đột logic <em>nội bộ</em>, nhưng ghép lại thì đơn hàng cũ sẽ hiển thị SAI giá nếu module Khuyến mãi vô tình dùng nhầm nguồn. Bạn chỉ ra mâu thuẫn này ngay ở giai đoạn review liên team, trước khi hai đội code xong mới phát hiện.</p>
<p><strong>Vì sao là mức ④:</strong> bạn soi được lỗi nằm ở <em>ranh giới giữa các hệ thống</em> — nơi mà mỗi tài liệu riêng lẻ đều &quot;đúng&quot;, chỉ sai khi ghép lại.</p>
