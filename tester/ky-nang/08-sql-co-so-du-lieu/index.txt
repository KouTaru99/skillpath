18:T17ea,<h1>SQL &amp; cơ sở dữ liệu sản phẩm</h1>
<p><strong>Định nghĩa.</strong> Tester cần đọc/ghi trực tiếp CSDL để: (1) <strong>chuẩn bị dữ liệu test</strong> không phụ thuộc UI (nhanh hơn nhiều lần), (2) <strong>xác minh dữ liệu lưu đúng</strong> sau thao tác — không chỉ tin những gì UI hiển thị (UI có thể hiện đúng mà DB lưu sai, hoặc ngược lại). 4 câu lệnh nền: <code>SELECT</code> (truy vấn), <code>INSERT</code> (thêm), <code>UPDATE</code> (sửa), <code>DELETE</code> (xoá). Nguyên tắc vàng của Tester dùng SQL: <strong>luôn kiểm chứng ở tầng dữ liệu</strong>, vì bug nguy hiểm nhất là loại UI báo &quot;thành công&quot; nhưng DB không lưu.</p>
<h2>▸ Junior — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Viết câu lệnh SQL cơ bản trên bảng đơn để tạo/sửa/xoá/truy vấn dữ liệu test — và biết dùng nó để kiểm chứng thay vì chỉ tin UI.</p>
<p><strong>Tình huống thực tế — test khoá tài khoản bằng SQL thay vì bấm tay 5 lần.</strong> Đăng nhập sai 5 lần trên giao diện vừa chậm vừa dễ sai thao tác. Bạn dựng thẳng trạng thái test bằng SQL rồi kiểm chứng kết quả cũng bằng SQL:</p>
<pre><code class="language-sql"><span class="hljs-comment">-- 1) DỰNG dữ liệu test: 5 lần đăng nhập sai của cùng 1 IP</span>
<span class="hljs-keyword">INSERT INTO</span> login_attempts (username, ip, success, created_at)
<span class="hljs-keyword">VALUES</span> (<span class="hljs-string">&#x27;admin&#x27;</span>, <span class="hljs-string">&#x27;10.0.0.5&#x27;</span>, <span class="hljs-literal">false</span>, NOW()),
       (<span class="hljs-string">&#x27;admin&#x27;</span>, <span class="hljs-string">&#x27;10.0.0.5&#x27;</span>, <span class="hljs-literal">false</span>, NOW()),
       (<span class="hljs-string">&#x27;admin&#x27;</span>, <span class="hljs-string">&#x27;10.0.0.5&#x27;</span>, <span class="hljs-literal">false</span>, NOW()),
       (<span class="hljs-string">&#x27;admin&#x27;</span>, <span class="hljs-string">&#x27;10.0.0.5&#x27;</span>, <span class="hljs-literal">false</span>, NOW()),
       (<span class="hljs-string">&#x27;admin&#x27;</span>, <span class="hljs-string">&#x27;10.0.0.5&#x27;</span>, <span class="hljs-literal">false</span>, NOW());

<span class="hljs-comment">-- 2) KÍCH hệ thống xử lý (hoặc đợi job chạy), rồi KIỂM CHỨNG ở DB:</span>
<span class="hljs-keyword">SELECT</span> username, is_locked, locked_until
<span class="hljs-keyword">FROM</span> accounts
<span class="hljs-keyword">WHERE</span> username <span class="hljs-operator">=</span> <span class="hljs-string">&#x27;admin&#x27;</span>;
<span class="hljs-comment">-- Mong đợi: is_locked = true, locked_until = NOW() + 15 phút</span>
</code></pre>
<p>Điểm ăn tiền của Junior giỏi SQL: khi UI báo &quot;tài khoản đã khoá&quot; mà câu <code>SELECT</code> cho thấy <code>is_locked</code> vẫn <code>false</code>, bạn bắt được một bug mà người chỉ nhìn giao diện sẽ bỏ qua hoàn toàn — UI hiển thị lạc quan còn DB mới là sự thật.</p>
<p><strong>Vì sao là mức ①:</strong> thao tác đúng trên bảng đơn và biết kiểm chứng bằng dữ liệu; chưa cần truy vấn quan hệ nhiều bảng.</p>
<h2>▸ Ex·V2 — ② Biết làm</h2>
<p><strong>Khác Junior:</strong> hiểu <strong>cấu trúc quan hệ nhiều bảng</strong> của sản phẩm, viết được truy vấn nối bảng (JOIN) để kiểm chứng logic mà UI không thể hiện hết.</p>
<p><strong>Tình huống thực tế — xác minh tính năng &quot;gộp cảnh báo thành sự cố&quot; lưu đúng quan hệ.</strong> UI chỉ hiện &quot;3 cảnh báo → 1 sự cố&quot;, nhưng bạn cần chắc quan hệ được lưu đúng ở tầng dữ liệu (nếu sai, báo cáo tuân thủ sau này đếm sai):</p>
<pre><code class="language-sql"><span class="hljs-comment">-- Kiểm chứng: mỗi sự cố gộp ĐÚNG số cảnh báo, không sót/không thừa</span>
<span class="hljs-keyword">SELECT</span> i.id            <span class="hljs-keyword">AS</span> su_co,
       i.severity      <span class="hljs-keyword">AS</span> muc_do,
       <span class="hljs-built_in">COUNT</span>(ia.alert_id) <span class="hljs-keyword">AS</span> so_canh_bao_gop
<span class="hljs-keyword">FROM</span> incidents i
<span class="hljs-keyword">JOIN</span> incident_alerts ia <span class="hljs-keyword">ON</span> ia.incident_id <span class="hljs-operator">=</span> i.id   <span class="hljs-comment">-- bảng trung gian</span>
<span class="hljs-keyword">WHERE</span> i.created_at <span class="hljs-operator">&gt;</span> NOW() <span class="hljs-operator">-</span> <span class="hljs-type">INTERVAL</span> <span class="hljs-string">&#x27;1 hour&#x27;</span>
<span class="hljs-keyword">GROUP</span> <span class="hljs-keyword">BY</span> i.id, i.severity
<span class="hljs-keyword">HAVING</span> <span class="hljs-built_in">COUNT</span>(ia.alert_id) <span class="hljs-operator">&lt;&gt;</span> <span class="hljs-number">3</span>;   <span class="hljs-comment">-- lọc ra sự cố gộp SAI số lượng</span>
<span class="hljs-comment">-- Kết quả rỗng = mọi sự cố đều gộp đúng 3 cảnh báo như thiết kế</span>
</code></pre>
<p>Mệnh đề <code>HAVING ... &lt;&gt; 3</code> biến truy vấn thành một &quot;cái bẫy bug&quot;: nó chỉ trả về dòng khi có sự cố gộp sai. Đây là tư duy Tester nâng cao — dùng SQL không chỉ để <em>xem</em> dữ liệu mà để <em>lọc thẳng ra chỗ sai</em>. Một câu SQL kiểm được điều mà 50 lần click UI không chắc bắt nổi.</p>
<p><strong>Vì sao là mức ②:</strong> truy vấn được quan hệ nhiều bảng đúng bản chất CSDL sản phẩm, và biến SQL thành công cụ săn bug — không chỉ thao tác trên bảng cô lập.</p>
