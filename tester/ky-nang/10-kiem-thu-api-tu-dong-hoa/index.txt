18:T1756,<h1>Kiểm thử API &amp; công cụ tự động hoá</h1>
<p><strong>Định nghĩa.</strong> Thay vì test qua giao diện (chậm, dễ sót), Tester dùng công cụ gọi thẳng API (như <strong>Postman</strong>) để kiểm chứng logic backend độc lập với UI, và dùng <strong>công cụ/framework tự động hoá có sẵn</strong> (Selenium/Playwright cho web, Appium cho mobile, JMeter cho tải) để chạy lại hàng loạt test case không cần thao tác tay. Với Tester ở mức này, trọng tâm là <strong>dùng công cụ có sẵn cho hiệu quả</strong> — chưa cần tự xây framework từ đầu (việc của cấp cao hơn).</p>
<h2>▸ Ex·V2 — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Dùng công cụ gọi API có sẵn (Postman) để test một endpoint theo mẫu — kiểm chứng cả luồng đúng lẫn luồng lỗi.</p>
<p><strong>Tình huống thực tế — test API tạo cảnh báo bằng Postman.</strong> API <code>POST /api/alerts</code> (tạo cảnh báo thủ công, dùng khi diễn tập). Bạn test cả hai mặt:</p>
<pre><code>POSTMAN — POST /api/alerts

  CA THUẬN (dữ liệu hợp lệ):
    Body: { &quot;severity&quot;: &quot;high&quot;, &quot;source_ip&quot;: &quot;192.168.1.1&quot; }
    → Mong đợi: 201 Created + response có &quot;id&quot; mới sinh
      + &quot;created_at&quot; gần thời điểm hiện tại

  CA LỖI (thiếu trường bắt buộc):
    Body: { &quot;source_ip&quot;: &quot;192.168.1.1&quot; }   ← thiếu severity
    → Mong đợi: 400 Bad Request + message rõ &quot;severity is required&quot;
      (KHÔNG phải 500 — thiếu input là lỗi lường trước được)

  CA LỖI (giá trị lạ):
    Body: { &quot;severity&quot;: &quot;SUPER_CRITICAL&quot;, ... }  ← enum không hợp lệ
    → Mong đợi: 400 + báo giá trị không thuộc {low, medium, high}
</code></pre>
<p>Test thẳng API bắt được điều test qua UI khó thấy: giao diện thường chỉ cho chọn enum từ dropdown (không bao giờ gửi được giá trị lạ), nhưng kẻ tấn công gọi thẳng API thì gửi được — nên ca &quot;giá trị lạ&quot; là bắt buộc với sản phẩm an ninh.</p>
<p><strong>Vì sao là mức ①:</strong> dùng đúng công cụ có sẵn cho ca đơn theo mẫu; chưa tự dựng bộ test chạy lặp tự động.</p>
<h2>▸ Ex·V3 — ② Biết làm</h2>
<p><strong>Khác V1:</strong> dùng công cụ/framework để <strong>tự động hoá</strong> — gom test rời thành bộ chạy lại được nhiều lần, gắn assertion (kiểm tra tự động) để không phải mắt người soi từng response.</p>
<p><strong>Tình huống thực tế 1 — Postman Collection có assertion, chạy mỗi build.</strong> Bạn gom các API test thủ công thành một collection có kiểm tra tự động bằng script:</p>
<pre><code class="language-javascript"><span class="hljs-comment">// Postman — tab &quot;Tests&quot; của request POST /api/alerts (ca thuận)</span>
pm.<span class="hljs-title function_">test</span>(<span class="hljs-string">&quot;Status 201&quot;</span>, <span class="hljs-function">() =&gt;</span> pm.<span class="hljs-property">response</span>.<span class="hljs-property">to</span>.<span class="hljs-property">have</span>.<span class="hljs-title function_">status</span>(<span class="hljs-number">201</span>));
pm.<span class="hljs-title function_">test</span>(<span class="hljs-string">&quot;Có id mới&quot;</span>, <span class="hljs-function">() =&gt;</span> {
  <span class="hljs-keyword">const</span> body = pm.<span class="hljs-property">response</span>.<span class="hljs-title function_">json</span>();
  pm.<span class="hljs-title function_">expect</span>(body.<span class="hljs-property">id</span>).<span class="hljs-property">to</span>.<span class="hljs-property">be</span>.<span class="hljs-title function_">a</span>(<span class="hljs-string">&quot;number&quot;</span>);
  pm.<span class="hljs-title function_">expect</span>(body.<span class="hljs-property">severity</span>).<span class="hljs-property">to</span>.<span class="hljs-title function_">eql</span>(<span class="hljs-string">&quot;high&quot;</span>);
});
pm.<span class="hljs-title function_">test</span>(<span class="hljs-string">&quot;Phản hồi &lt; 500ms&quot;</span>, <span class="hljs-function">() =&gt;</span> pm.<span class="hljs-title function_">expect</span>(pm.<span class="hljs-property">response</span>.<span class="hljs-property">responseTime</span>).<span class="hljs-property">to</span>.<span class="hljs-title function_">below</span>(<span class="hljs-number">500</span>));
</code></pre>
<p>Chạy cả collection (20 request) mỗi khi có build mới — 30 giây thay cho 30 phút test tay, và không bao giờ &quot;quên&quot; case nào.</p>
<p><strong>Tình huống thực tế 2 — kịch bản Selenium cho luồng UI lặp lại.</strong> Với phần giao diện, bạn viết kịch bản tự động cho luồng smoke hay phải chạy đi chạy lại:</p>
<pre><code>Kịch bản Selenium &quot;smoke đăng nhập + xem cảnh báo&quot;:
  1. Mở /login → điền tài khoản qa_auto → submit
  2. Chờ dashboard load → khẳng định URL = /dashboard
  3. Vào /alerts → khẳng định bảng có ≥1 dòng
  4. Khẳng định cột &quot;mức độ&quot; của dòng đầu hiển thị đúng badge màu
→ Chạy tự động mỗi sáng trước khi Tester bắt đầu test tay:
  build &quot;sống&quot; thì mới đáng test sâu.
</code></pre>
<p><strong>Vì sao là mức ②:</strong> dùng công cụ tự động hoá có sẵn để tăng tốc kiểm thử lặp lại và biến &quot;test tay tốn công&quot; thành &quot;chạy một nút&quot;; chưa cần tự thiết kế framework tự động hoá từ nền.</p>
