18:T1208,<h1>Git/SVN</h1>
<p><strong>Định nghĩa.</strong> Công cụ quản lý phiên bản mã nguồn — ghi lại lịch sử thay đổi, cho phép nhiều người cùng sửa code mà không ghi đè lẫn nhau. <strong>Git</strong> phổ biến nhất hiện nay (phân tán — mỗi máy có bản sao đầy đủ lịch sử); <strong>SVN</strong> cũ hơn (tập trung — một server giữ lịch sử chính).</p>
<h2>▸ Entry — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Dùng được các lệnh Git cơ bản: commit, push, pull — theo đúng quy trình của team.</p>
<p><strong>Ví dụ thực tế — vòng lặp làm việc cơ bản mỗi ngày.</strong></p>
<pre><code class="language-bash">git pull                          <span class="hljs-comment"># lấy code mới nhất từ team trước khi bắt đầu</span>
<span class="hljs-comment"># ... sửa code ...</span>
git add src/OrderService.java
git commit -m <span class="hljs-string">&quot;Thêm API lấy đơn hàng theo trạng thái&quot;</span>
git push                          <span class="hljs-comment"># đẩy thay đổi lên để team khác thấy</span>
</code></pre>
<p>Bạn commit với message rõ ràng (nói được &quot;làm gì&quot;, không phải &quot;fix bug&quot; chung chung), và luôn <code>pull</code> trước khi bắt đầu để tránh code của mình dựa trên bản cũ.</p>
<p><strong>Vì sao là mức ①:</strong> dùng đúng lệnh cơ bản trong quy trình đơn giản (1 nhánh) — chưa xử lý xung đột merge hay làm việc với nhiều nhánh song song.</p>
<h2>▸ Ex·V1 — ② Biết làm</h2>
<p><strong>Khác Entry:</strong> thành thạo chia nhánh (branch) và merge, xử lý được xung đột (conflict) đơn giản.</p>
<p><strong>Ví dụ thực tế — làm việc trên nhánh riêng rồi merge vào nhánh chính.</strong></p>
<pre><code class="language-bash">git checkout -b feature/order-discount    <span class="hljs-comment"># tạo nhánh riêng cho tính năng</span>
<span class="hljs-comment"># ... code + commit ...</span>
git checkout main
git merge feature/order-discount          <span class="hljs-comment"># gộp lại vào nhánh chính</span>
</code></pre>
<pre><code># Nếu có conflict (2 người cùng sửa 1 dòng), Git đánh dấu:
&lt;&lt;&lt;&lt;&lt;&lt;&lt; HEAD
double total = price * 0.9;
=======
double total = price - discount;
&gt;&gt;&gt;&gt;&gt;&gt;&gt; feature/order-discount
</code></pre>
<p>Bạn tự đọc và chọn giữ phần đúng (hoặc gộp cả 2 ý), xoá các dấu <code>&lt;&lt;&lt;&lt;&lt;&lt;&lt;</code>/<code>=======</code>/<code>&gt;&gt;&gt;&gt;&gt;&gt;&gt;</code>, rồi commit lại — không hoảng khi thấy conflict.</p>
<p><strong>Vì sao là mức ②:</strong> làm việc được với nhiều nhánh song song, tự xử lý xung đột đơn giản — chưa dùng kỹ thuật nâng cao (rebase, cherry-pick).</p>
<h2>▸ Specialist·V1 — ③ Thành thạo</h2>
<p><strong>Khác Ex·V1:</strong> thành thạo version control nâng cao — merge phức tạp, <strong>rebase</strong>, xem <strong>reflog</strong>, <strong>cherry-pick</strong>, dùng <strong>patch</strong>.</p>
<p><strong>Ví dụ thực tế — cherry-pick một fix khẩn cấp sang nhánh release mà không mang theo các commit khác.</strong></p>
<pre><code class="language-bash">git <span class="hljs-built_in">log</span> feature/big-refactor --oneline   <span class="hljs-comment"># tìm commit fix cụ thể, vd a1b2c3d</span>
git checkout release/v2.1
git cherry-pick a1b2c3d                   <span class="hljs-comment"># chỉ lấy đúng 1 commit đó, không mang cả nhánh</span>
</code></pre>
<pre><code class="language-bash"><span class="hljs-comment"># Lỡ tay reset nhầm, mất commit? reflog vẫn giữ lịch sử thao tác (không phải lịch sử code)</span>
git reflog                                <span class="hljs-comment"># thấy được cả commit &quot;đã mất&quot; trước khi reset</span>
git reset --hard HEAD@{2}                 <span class="hljs-comment"># khôi phục lại</span>
</code></pre>
<p>Bạn dùng <code>reflog</code> như &quot;lưới an toàn&quot; cuối cùng khi thao tác Git sai — Git hiếm khi thực sự XOÁ dữ liệu ngay, chỉ là bạn chưa biết cách tìm lại.</p>
<p><strong>Vì sao là mức ③:</strong> bạn dùng thành thạo các công cụ Git nâng cao để xử lý tình huống phức tạp — không chỉ merge/conflict cơ bản.</p>
