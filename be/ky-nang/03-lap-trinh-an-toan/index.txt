18:T1b32,<h1>Lập trình an toàn (Secure Coding)</h1>
<p><strong>Định nghĩa.</strong> Viết code phòng ngừa lỗ hổng bảo mật ngay từ khâu lập trình. Tham chiếu phổ biến là <strong>OWASP Top 10</strong> (injection, hỏng xác thực/phân quyền, lộ dữ liệu nhạy cảm...). Ở phía Back-end, trọng tâm: chống <strong>SQL injection</strong>, <strong>hash mật khẩu</strong> đúng cách (không lưu plain text), không lộ <strong>secret</strong> (connection string, API key) trong code/log. Với công ty an ninh mạng, đây là kỹ năng bắt buộc xuyên suốt mọi level.</p>
<h2>▸ Entry — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Biết các lỗ hổng phổ biến, không hardcode secret trong code, dùng tham số hoá câu SQL (không nối chuỗi trực tiếp), hash mật khẩu thay vì lưu plain text.</p>
<p><strong>Ví dụ thực tế — chống SQL injection + hash mật khẩu (Java/Spring).</strong></p>
<pre><code class="language-java"><span class="hljs-comment">// ❌ NGUY HIỂM: nối chuỗi trực tiếp → kẻ xấu truyền username = &quot;&#x27; OR &#x27;1&#x27;=&#x27;1&quot; là vào được</span>
<span class="hljs-type">String</span> <span class="hljs-variable">sql</span> <span class="hljs-operator">=</span> <span class="hljs-string">&quot;SELECT * FROM users WHERE username = &#x27;&quot;</span> + username + <span class="hljs-string">&quot;&#x27;&quot;</span>;

<span class="hljs-comment">// ✅ AN TOÀN: tham số hoá (JPA/Spring Data tự làm điều này khi dùng đúng cách)</span>
<span class="hljs-meta">@Query(&quot;SELECT u FROM User u WHERE u.username = :username&quot;)</span>
User <span class="hljs-title function_">findByUsername</span><span class="hljs-params">(<span class="hljs-meta">@Param(&quot;username&quot;)</span> String username)</span>;
</code></pre>
<pre><code class="language-java"><span class="hljs-comment">// ❌ NGUY HIỂM: lưu mật khẩu dạng thường</span>
user.setPassword(rawPassword);

<span class="hljs-comment">// ✅ AN TOÀN: hash bằng BCrypt trước khi lưu — dù lộ DB, mật khẩu vẫn không đọc được</span>
user.setPassword(passwordEncoder.encode(rawPassword));
</code></pre>
<p><strong>Vì sao là mức ①:</strong> nhận ra rủi ro và tránh lỗi cơ bản, chưa áp dụng nhất quán toàn luồng.</p>
<h2>▸ Ex·V1 — ② Biết làm</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Áp dụng nhất quán trong mọi API — kiểm quyền ở backend (không tin FE), validate input đầy đủ, không lộ thông tin nhạy cảm qua thông báo lỗi.</p>
<p><strong>Ví dụ thực tế — kiểm quyền ở backend, không chỉ dựa vào FE ẩn nút.</strong></p>
<pre><code class="language-java"><span class="hljs-meta">@DeleteMapping(&quot;/api/orders/{id}&quot;)</span>
<span class="hljs-keyword">public</span> <span class="hljs-keyword">void</span> <span class="hljs-title function_">deleteOrder</span><span class="hljs-params">(<span class="hljs-meta">@PathVariable</span> Long id, Authentication auth)</span> {
  <span class="hljs-type">Order</span> <span class="hljs-variable">order</span> <span class="hljs-operator">=</span> orderRepository.findById(id).orElseThrow();
  <span class="hljs-keyword">if</span> (!order.getCustomerId().equals(getCurrentUserId(auth))) {
    <span class="hljs-keyword">throw</span> <span class="hljs-keyword">new</span> <span class="hljs-title class_">AccessDeniedException</span>(<span class="hljs-string">&quot;Không có quyền xoá đơn của người khác&quot;</span>);
  }
  orderRepository.delete(order);
}
</code></pre>
<p>Dù FE có ẩn nút &quot;Xoá&quot; khi không phải chủ đơn, backend vẫn PHẢI tự kiểm — vì ai đó có thể gọi thẳng API bằng Postman, bỏ qua giao diện.</p>
<p><strong>Vì sao là mức ②:</strong> bảo mật thành thói quen trong từng API, không chỉ tránh lỗi rõ ràng.</p>
<h2>▸ Ex·V3 — ③ Thành thạo</h2>
<p><strong>Khác V1:</strong> nắm vững lập trình an toàn ở tầm hệ thống — đề xuất và triển khai giải pháp bảo mật, không chỉ tự phòng thủ trong code của mình.</p>
<p><strong>Ví dụ thực tế — đề xuất rate limiting để chống brute-force đăng nhập.</strong> Bạn nhận thấy API <code>/api/login</code> không giới hạn số lần thử — kẻ xấu có thể dò mật khẩu bằng cách gọi liên tục. Bạn đề xuất và triển khai:</p>
<pre><code class="language-java"><span class="hljs-meta">@RateLimiter(name = &quot;login&quot;, fallbackMethod = &quot;loginFallback&quot;)</span>
<span class="hljs-meta">@PostMapping(&quot;/api/login&quot;)</span>
<span class="hljs-keyword">public</span> TokenResponse <span class="hljs-title function_">login</span><span class="hljs-params">(<span class="hljs-meta">@RequestBody</span> LoginRequest req)</span> { ... }

<span class="hljs-keyword">public</span> TokenResponse <span class="hljs-title function_">loginFallback</span><span class="hljs-params">(LoginRequest req, Throwable t)</span> {
  <span class="hljs-keyword">throw</span> <span class="hljs-keyword">new</span> <span class="hljs-title class_">TooManyRequestsException</span>(<span class="hljs-string">&quot;Thử lại sau ít phút&quot;</span>);
}
</code></pre>
<p>Giới hạn số lần gọi <code>/api/login</code> từ cùng một IP trong một khoảng thời gian — chặn kiểu tấn công dò mật khẩu hàng loạt, một lớp phòng thủ mà từng API riêng lẻ không tự có.</p>
<p><strong>Vì sao là mức ③:</strong> bạn chủ động đề xuất giải pháp bảo mật ở tầm hệ thống — không chỉ viết code an toàn cho phần việc của mình.</p>
<h2>▸ Senior·V3 — ④ Chuyên sâu</h2>
<p><strong>Khác Ex·V3:</strong> đánh giá rủi ro bảo mật <strong>trước khi code</strong> (threat modeling) ngay từ khâu thiết kế, không chỉ review sau khi đã có code.</p>
<p><strong>Ví dụ thực tế — threat modeling cho API &quot;chia sẻ đơn hàng qua link công khai&quot;.</strong> Trước khi ai viết dòng code nào, bạn đặt câu hỏi phòng thủ: Link có đoán được không (nếu dùng <code>orderId=123</code> tuần tự → kẻ xấu dò hết đơn người khác, cần đổi sang token ngẫu nhiên)? Link có hết hạn không? Ai xem link có thấy thông tin nhạy cảm (địa chỉ đầy đủ) không? Ba câu hỏi này được đặt ra khi sửa còn rẻ — phát hiện muộn (sau khi đã có schema, dữ liệu thật) sẽ tốn công hơn nhiều.</p>
<p><strong>Vì sao là mức ④:</strong> bạn chủ động đánh giá rủi ro bảo mật ở khâu thiết kế — mức phòng thủ sâu nhất, rẻ nhất.</p>
