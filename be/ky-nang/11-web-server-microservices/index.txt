18:T18ec,<h1>Web server / Microservices cơ bản</h1>
<p><strong>Định nghĩa.</strong> <strong>Web server</strong> (Tomcat, Nginx) nhận request HTTP và trả response — Spring Boot có sẵn Tomcat nhúng bên trong nên bạn không cần cài đặt riêng. <strong>Microservices</strong> là kiến trúc chia hệ thống lớn thành nhiều service nhỏ, mỗi service lo một nghiệp vụ, giao tiếp qua API — khác với <strong>monolith</strong> (mọi thứ gộp trong một ứng dụng duy nhất).</p>
<h2>▸ Ex·V1 — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Hiểu web server nhận/trả request thế nào, hiểu khái niệm microservices khác monolith ra sao.</p>
<p><strong>Ví dụ thực tế — một request đi qua web server tới code của bạn.</strong></p>
<pre><code>Client → [Web server: Tomcat nhúng trong Spring Boot] → [Controller của bạn] → Response
</code></pre>
<pre><code class="language-java"><span class="hljs-meta">@RestController</span>
<span class="hljs-keyword">public</span> <span class="hljs-keyword">class</span> <span class="hljs-title class_">OrderController</span> {
  <span class="hljs-meta">@GetMapping(&quot;/api/orders/{id}&quot;)</span>     <span class="hljs-comment">// Tomcat nhận request GET, chuyển tới hàm này</span>
  <span class="hljs-keyword">public</span> Order <span class="hljs-title function_">getOrder</span><span class="hljs-params">(<span class="hljs-meta">@PathVariable</span> Long id)</span> { ... }
}
</code></pre>
<p>Bạn hiểu Tomcat (web server) là lớp nhận request đầu tiên, chuyển đúng tới hàm xử lý (<code>@GetMapping</code>) dựa trên URL — không cần tự viết code &quot;nghe&quot; cổng mạng.</p>
<p><strong>Ví dụ 2 — phân biệt monolith vs microservices.</strong></p>
<pre><code>Monolith: 1 ứng dụng Spring Boot xử lý cả đơn hàng, thanh toán, thông báo.
Microservices: order-service, payment-service, notification-service —
               3 ứng dụng độc lập, giao tiếp qua API.
</code></pre>
<p><strong>Vì sao là mức ①:</strong> hiểu khái niệm nền tảng — chưa tự thiết kế hay vận hành một hệ microservices thật.</p>
<h2>▸ Ex·V3 — ② Biết làm</h2>
<p><strong>Khác V1:</strong> nắm vững công nghệ/kiến trúc Docker Container, Container Orchestration (điều phối nhiều container), và hiểu các mô hình Cloud (IaaS/PaaS/SaaS, Cloud Native App).</p>
<p><strong>Ví dụ thực tế — phân biệt IaaS/PaaS khi chọn hạ tầng triển khai.</strong></p>
<pre><code>IaaS (Infrastructure as a Service): thuê máy ảo trần — TỰ cài OS, runtime, mọi thứ.
  Ví dụ: EC2 (AWS). Kiểm soát nhiều, nhưng tự quản lý nhiều việc.

PaaS (Platform as a Service): chỉ cần đẩy code/jar lên, nền tảng tự lo server, scaling.
  Ví dụ: Heroku, Google App Engine. Ít việc quản lý hơn, đổi lại ít tuỳ biến hạ tầng hơn.
</code></pre>
<p>Bạn hiểu đây là lựa chọn đánh đổi giữa <strong>kiểm soát</strong> và <strong>tiện lợi</strong> — không có cái nào &quot;tốt hơn&quot; tuyệt đối, chọn theo nhu cầu dự án.</p>
<p><strong>Vì sao là mức ②:</strong> bạn hiểu được các mô hình triển khai và điều phối container ở mức khái niệm vận hành — chưa tự thiết kế một hệ microservices hoàn chỉnh (việc đó ở level Senior).</p>
<h2>▸ Senior·V1 — ③ Thành thạo</h2>
<p><strong>Khác Ex·V3:</strong> tự thiết kế được <strong>ranh giới service</strong> (service boundary) — mỗi service nên ôm nghiệp vụ gì, và dùng <strong>API Gateway</strong> làm cửa ngõ chung.</p>
<p><strong>Ví dụ thực tế — vẽ ranh giới service theo nghiệp vụ, không theo tầng kỹ thuật.</strong></p>
<pre><code>❌ Chia theo tầng kỹ thuật: &quot;controller-service&quot;, &quot;database-service&quot;
   → mỗi thay đổi nghiệp vụ đều phải sửa cả 2 service, mất ý nghĩa &quot;độc lập&quot;.

✅ Chia theo nghiệp vụ: order-service (toàn bộ vòng đời đơn hàng),
   payment-service (toàn bộ vòng đời thanh toán) — mỗi service tự chứa
   đủ logic + dữ liệu cho MỘT nghiệp vụ trọn vẹn.
</code></pre>
<pre><code>Client → [API Gateway] → order-service / payment-service / notification-service
          (1 cửa ngõ duy nhất: xác thực, rate limit, định tuyến — client không
           cần biết có bao nhiêu service phía sau)
</code></pre>
<p>Bạn hiểu ranh giới đúng (theo nghiệp vụ) giúp mỗi service thay đổi độc lập; API Gateway giúp client không phải gọi trực tiếp từng service riêng lẻ.</p>
<p><strong>Vì sao là mức ③:</strong> bạn tự thiết kế được ranh giới service hợp lý — không chỉ hiểu khái niệm microservices/container ở mức vận hành.</p>
<h2>▸ Senior·V2 — ④ Chuyên sâu</h2>
<p><strong>Khác Senior·V1:</strong> xử lý được vấn đề <strong>giao dịch xuyên nhiều service</strong> — khi một nghiệp vụ cần vài service cùng thành công hoặc cùng thất bại, mà không có transaction chung như trong 1 CSDL đơn.</p>
<p><strong>Ví dụ thực tế — Saga pattern cho nghiệp vụ &quot;đặt hàng&quot; chạm 3 service.</strong></p>
<pre><code>1. order-service: tạo đơn (PENDING)
2. inventory-service: trừ kho — nếu THẤT BẠI → gửi sự kiện huỷ ngược lại bước 1
3. payment-service: trừ tiền — nếu THẤT BẠI → gửi sự kiện hoàn kho (bước 2) + huỷ đơn (bước 1)
</code></pre>
<p>Không có transaction chung xuyên 3 service (khác với <code>@Transactional</code> trong 1 CSDL), bạn thiết kế mỗi bước có <strong>hành động bù trừ</strong> (compensating action) để hoàn tác nếu bước sau thất bại — đây là bản chất Saga pattern, khác hẳn transaction truyền thống.</p>
<p><strong>Vì sao là mức ④:</strong> bạn xử lý được bài toán toàn vẹn dữ liệu xuyên nhiều service — mức cao nhất của kỹ năng thiết kế microservices trong thang Senior.</p>
