18:T1e7c,<h1>Cơ sở dữ liệu (SQL &amp; NoSQL) — thiết kế &amp; tối ưu</h1>
<p><strong>Định nghĩa.</strong> Khác Front-end (chỉ cần hiểu đủ để phối hợp), Back-end <strong>tự tay thiết kế và tối ưu</strong> CSDL. <strong>SQL</strong> (MySQL, PostgreSQL) lưu dữ liệu dạng bảng có quan hệ, đảm bảo tính toàn vẹn (ràng buộc khoá ngoại, transaction). <strong>NoSQL</strong> (MongoDB, Redis) lưu dạng linh hoạt hơn (tài liệu, key-value), đánh đổi lấy tốc độ/khả năng mở rộng khi dữ liệu không cần quan hệ chặt.</p>
<h2>▸ Entry — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Hiểu khác biệt SQL vs NoSQL, thiết kế được bảng đơn giản với khoá chính/khoá ngoại, viết được câu SQL CRUD cơ bản.</p>
<p><strong>Ví dụ thực tế — thiết kế bảng đơn hàng có quan hệ, dùng JPA (Java).</strong></p>
<pre><code class="language-java"><span class="hljs-meta">@Entity</span>
<span class="hljs-keyword">public</span> <span class="hljs-keyword">class</span> <span class="hljs-title class_">Order</span> {
  <span class="hljs-meta">@Id</span> <span class="hljs-meta">@GeneratedValue</span> Long id;
  String status;
  <span class="hljs-meta">@ManyToOne</span>
  <span class="hljs-meta">@JoinColumn(name = &quot;customer_id&quot;)</span>
  Customer customer;          <span class="hljs-comment">// khoá ngoại → mỗi đơn thuộc về 1 khách hàng</span>
}
</code></pre>
<pre><code class="language-sql"><span class="hljs-comment">-- Tương đương SQL thuần</span>
<span class="hljs-keyword">CREATE TABLE</span> orders (
  id SERIAL <span class="hljs-keyword">PRIMARY KEY</span>,
  status <span class="hljs-type">VARCHAR</span>(<span class="hljs-number">20</span>),
  customer_id <span class="hljs-type">INT</span> <span class="hljs-keyword">REFERENCES</span> customers(id)
);
</code></pre>
<p>Bạn hiểu vì sao dùng khoá ngoại (<code>customer_id REFERENCES customers(id)</code>) thay vì lưu lặp lại tên khách hàng ở mỗi đơn — tránh trùng lặp, tránh lệch dữ liệu khi khách đổi tên.</p>
<p><strong>Vì sao là mức ①:</strong> thiết kế và truy vấn được CSDL cho bài toán đơn giản, chưa tối ưu hiệu năng hay xử lý quy mô lớn.</p>
<h2>▸ Ex·V1 — ② Biết làm</h2>
<p><strong>Khác Entry:</strong> thiết kế được <strong>index</strong>, hiểu <strong>partition</strong> cơ bản, tối ưu câu lệnh SQL đơn giản.</p>
<p><strong>Ví dụ thực tế — thêm index để tăng tốc truy vấn tìm đơn theo khách hàng.</strong></p>
<pre><code class="language-sql"><span class="hljs-comment">-- Trước: quét toàn bảng để tìm đơn của 1 khách hàng — chậm khi bảng lớn</span>
<span class="hljs-keyword">SELECT</span> <span class="hljs-operator">*</span> <span class="hljs-keyword">FROM</span> orders <span class="hljs-keyword">WHERE</span> customer_id <span class="hljs-operator">=</span> <span class="hljs-number">42</span>;

<span class="hljs-comment">-- Thêm index cho cột hay lọc:</span>
<span class="hljs-keyword">CREATE</span> INDEX idx_orders_customer <span class="hljs-keyword">ON</span> orders(customer_id);
</code></pre>
<p>Bạn hiểu index giống mục lục sách — giúp CSDL tìm nhanh mà không phải đọc hết từng trang (dòng), nhưng cũng biết đánh đổi: mỗi index thêm làm <code>INSERT</code>/<code>UPDATE</code> chậm đi một chút.</p>
<p><strong>Vì sao là mức ②:</strong> tự thiết kế index và tối ưu câu lệnh SQL cơ bản — chưa xử lý CSDL ở quy mô lớn hay dùng các kỹ thuật nâng cao (view, stored procedure).</p>
<h2>▸ Ex·V2 — ③ Thành thạo</h2>
<p><strong>Khác V1:</strong> thiết kế được CSDL đã <strong>chuẩn hoá</strong> (normalize) đúng cách, dùng <strong>view/stored procedure/trigger</strong>, phân biệt được <strong>clustered vs non-clustered index</strong>.</p>
<p><strong>Ví dụ thực tế — trigger tự động cập nhật tồn kho khi có đơn hàng mới.</strong></p>
<pre><code class="language-sql"><span class="hljs-keyword">CREATE</span> <span class="hljs-keyword">TRIGGER</span> trg_update_stock
AFTER <span class="hljs-keyword">INSERT</span> <span class="hljs-keyword">ON</span> order_items
<span class="hljs-keyword">FOR</span> <span class="hljs-keyword">EACH</span> <span class="hljs-type">ROW</span>
<span class="hljs-keyword">BEGIN</span>
  <span class="hljs-keyword">UPDATE</span> products <span class="hljs-keyword">SET</span> stock <span class="hljs-operator">=</span> stock <span class="hljs-operator">-</span> NEW.quantity <span class="hljs-keyword">WHERE</span> id <span class="hljs-operator">=</span> NEW.product_id;
<span class="hljs-keyword">END</span>;
</code></pre>
<p>Trigger tự chạy mỗi khi có dòng <code>order_items</code> mới, đảm bảo tồn kho luôn được cập nhật ngay tại tầng CSDL — không phụ thuộc vào việc code Java có nhớ gọi hàm trừ kho hay không.</p>
<p><strong>Ví dụ 2 — clustered vs non-clustered index.</strong> Clustered index quyết định <strong>thứ tự vật lý</strong> dữ liệu lưu trên đĩa (mỗi bảng chỉ có 1); non-clustered index là một cấu trúc tra cứu riêng, trỏ ngược về dữ liệu gốc (một bảng có thể có nhiều). Bạn chọn cột làm clustered index (thường là khoá chính) dựa trên cột được truy vấn theo khoảng (range) nhiều nhất.</p>
<p><strong>Vì sao là mức ③:</strong> bạn thiết kế CSDL đúng chuẩn và dùng được công cụ nâng cao (view/trigger/stored procedure) — không chỉ viết câu lệnh SQL cơ bản.</p>
<h2>▸ Ex·V3 — ④ Chuyên sâu</h2>
<p><strong>Khác V2:</strong> phân tích/tối ưu hiệu năng dựa trên hiểu biết sâu về CSDL — biết cách dữ liệu được lưu trữ bên trong, cách index được lưu, cách CSDL được nhân bản (mirror/replicate), và <strong>2PC</strong> (Two-Phase Commit — giao thức đảm bảo nhiều CSDL cùng commit hoặc cùng rollback một giao dịch).</p>
<p><strong>Ví dụ thực tế — đọc execution plan để hiểu CSDL thực sự làm gì bên trong.</strong></p>
<pre><code class="language-sql">EXPLAIN ANALYZE <span class="hljs-keyword">SELECT</span> <span class="hljs-operator">*</span> <span class="hljs-keyword">FROM</span> orders <span class="hljs-keyword">WHERE</span> customer_id <span class="hljs-operator">=</span> <span class="hljs-number">42</span> <span class="hljs-keyword">AND</span> status <span class="hljs-operator">=</span> <span class="hljs-string">&#x27;PAID&#x27;</span>;
<span class="hljs-comment">-- Index Scan using idx_orders_customer (cost=0.29..8.31 rows=1)</span>
<span class="hljs-comment">--   Filter: status = &#x27;PAID&#x27;::text   ← lọc thêm SAU khi đã dùng index tìm customer_id</span>
</code></pre>
<p>Bạn nhận ra: index chỉ giúp tìm nhanh theo <code>customer_id</code>, còn điều kiện <code>status = 'PAID'</code> vẫn phải lọc thêm sau đó (không có index riêng cho nó) — nếu truy vấn này chạy thường xuyên, bạn cân nhắc tạo <strong>composite index</strong> trên cả 2 cột <code>(customer_id, status)</code> để CSDL tìm đúng ngay từ đầu, không cần lọc thêm bước sau.</p>
<p><strong>Vì sao là mức ④:</strong> bạn hiểu đủ sâu cơ chế bên trong CSDL để tối ưu hiệu năng và đảm bảo toàn vẹn dữ liệu ở tình huống phức tạp — mức cao nhất trong thang Entry→Experienced.</p>
