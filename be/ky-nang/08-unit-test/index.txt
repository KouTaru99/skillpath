18:T18a0,<h1>Unit Test</h1>
<p><strong>Định nghĩa.</strong> <strong>Unit test</strong> là bài kiểm thử tự động cho từng <em>đơn vị</em> nhỏ (một hàm, một service) — chạy độc lập, nhanh, khẳng định &quot;đầu vào X thì đầu ra phải Y&quot;. Mục đích: bắt lỗi sớm, cho phép sửa/refactor mà không sợ vỡ ngầm. Mẫu <strong>AAA</strong> (Arrange – Act – Assert). Java dùng <strong>JUnit</strong> (viết test) + <strong>Mockito</strong> (giả lập phụ thuộc).</p>
<h2>▸ Entry — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Viết test cho hàm/service đơn giản theo hướng dẫn, biết chạy và đọc kết quả pass/fail.</p>
<p><strong>Ví dụ thực tế — test một service tính giảm giá.</strong></p>
<pre><code class="language-java"><span class="hljs-keyword">class</span> <span class="hljs-title class_">DiscountServiceTest</span> {
  <span class="hljs-keyword">private</span> <span class="hljs-keyword">final</span> <span class="hljs-type">DiscountService</span> <span class="hljs-variable">service</span> <span class="hljs-operator">=</span> <span class="hljs-keyword">new</span> <span class="hljs-title class_">DiscountService</span>();

  <span class="hljs-meta">@Test</span>
  <span class="hljs-keyword">void</span> ápDụng10PhầnTrămChoĐơnTrên500K() {
    <span class="hljs-comment">// Arrange</span>
    <span class="hljs-type">double</span> <span class="hljs-variable">total</span> <span class="hljs-operator">=</span> <span class="hljs-number">600_000</span>;
    <span class="hljs-comment">// Act</span>
    <span class="hljs-type">double</span> <span class="hljs-variable">result</span> <span class="hljs-operator">=</span> service.applyDiscount(total);
    <span class="hljs-comment">// Assert</span>
    assertEquals(<span class="hljs-number">540_000</span>, result);   <span class="hljs-comment">// giảm 10% cho đơn &gt; 500k</span>
  }

  <span class="hljs-meta">@Test</span>
  <span class="hljs-keyword">void</span> khôngGiảmGiáChoĐơnDưới500K() {
    assertEquals(<span class="hljs-number">300_000</span>, service.applyDiscount(<span class="hljs-number">300_000</span>));   <span class="hljs-comment">// ca biên: không đủ điều kiện</span>
  }
}
</code></pre>
<p>Bạn chọn vài ca tiêu biểu (giá trị thường + biên) thay vì test bừa.</p>
<p><strong>Vì sao là mức ①:</strong> viết được test cơ bản theo mẫu AAA, chưa test được service có phụ thuộc (cần mock) hay luồng bất đồng bộ.</p>
<h2>▸ Ex·V1 — ② Biết làm</h2>
<p><strong>Khác Entry:</strong> hiểu cách <strong>thiết kế code để test được</strong> — tách phụ thuộc ra ngoài (dependency injection) để có thể mock khi test, thay vì code cứng khó test.</p>
<p><strong>Ví dụ thực tế — mock một dependency (Mockito) để test service không phụ thuộc CSDL thật.</strong></p>
<pre><code class="language-java"><span class="hljs-meta">@ExtendWith(MockitoExtension.class)</span>
<span class="hljs-keyword">class</span> <span class="hljs-title class_">OrderServiceTest</span> {
  <span class="hljs-meta">@Mock</span> OrderRepository repository;      <span class="hljs-comment">// giả lập, không chạm CSDL thật</span>
  <span class="hljs-meta">@InjectMocks</span> OrderService service;

  <span class="hljs-meta">@Test</span>
  <span class="hljs-keyword">void</span> trảLỗiKhiKhôngTìmThấyĐơn() {
    <span class="hljs-keyword">when</span>(repository.findById(<span class="hljs-number">1L</span>)).thenReturn(Optional.empty());
    assertThrows(OrderNotFoundException.class, () -&gt; service.getOrder(<span class="hljs-number">1L</span>));
  }
}
</code></pre>
<p>Test chạy nhanh (không cần CSDL thật) và ổn định (không phụ thuộc dữ liệu có sẵn) — nhờ <code>OrderService</code> nhận <code>OrderRepository</code> qua constructor (dependency injection) thay vì tự tạo bên trong.</p>
<p><strong>Vì sao là mức ②:</strong> thiết kế code để test được và mock đúng phụ thuộc — không chỉ test hàm thuần đơn giản.</p>
<h2>▸ Senior·V1 — ③ Thành thạo</h2>
<p><strong>Khác Ex·V2:</strong> setup được <strong>integration test</strong> — chạy với CSDL thật (không chỉ mock) để bắt lỗi ở tầng ghép nối mà unit test kèm mock không thấy được.</p>
<p><strong>Ví dụ thực tế — Testcontainers: chạy test với PostgreSQL thật trong container, tự dọn sau khi xong.</strong></p>
<pre><code class="language-java"><span class="hljs-meta">@Testcontainers</span>
<span class="hljs-keyword">class</span> <span class="hljs-title class_">OrderRepositoryIntegrationTest</span> {
  <span class="hljs-meta">@Container</span>
  <span class="hljs-keyword">static</span> PostgreSQLContainer&lt;?&gt; postgres = <span class="hljs-keyword">new</span> <span class="hljs-title class_">PostgreSQLContainer</span>&lt;&gt;(<span class="hljs-string">&quot;postgres:16&quot;</span>);

  <span class="hljs-meta">@Test</span>
  <span class="hljs-keyword">void</span> lưuVàTruyVấnĐơnHàngThậtTrênCSDL() {
    <span class="hljs-type">Order</span> <span class="hljs-variable">saved</span> <span class="hljs-operator">=</span> orderRepository.save(<span class="hljs-keyword">new</span> <span class="hljs-title class_">Order</span>(<span class="hljs-string">&quot;PENDING&quot;</span>));
    Optional&lt;Order&gt; found = orderRepository.findById(saved.getId());
    assertTrue(found.isPresent());   <span class="hljs-comment">// test với query THẬT, không phải mock giả định</span>
  }
}
</code></pre>
<p>Mock (<code>OrderRepository</code> giả) chỉ đảm bảo service gọi đúng hàm, nhưng KHÔNG bắt được lỗi trong câu query JPA thật (vd sai tên cột, sai kiểu dữ liệu). Testcontainers tự dựng CSDL thật trong Docker chỉ cho lúc test, tự dọn dẹp sau khi xong — không cần CSDL cài sẵn trên máy.</p>
<p><strong>Vì sao là mức ③:</strong> setup được integration test bắt đúng lớp lỗi mà unit test/mock không bắt được — không chỉ test logic thuần.</p>
