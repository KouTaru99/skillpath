18:T1b62,<h1>Hệ điều hành &amp; I/O</h1>
<p><strong>Định nghĩa.</strong> Ứng dụng Back-end chạy trên một hệ điều hành (Windows hoặc phổ biến hơn là <strong>Linux</strong> ở server) — hiểu tổng quan các tầng cơ bản (tiến trình, bộ nhớ, file system) giúp bạn hiểu ứng dụng của mình thực sự đang làm gì bên dưới lớp code. <strong>I/O</strong> (Input/Output) là việc đọc/ghi dữ liệu ra ngoài chương trình — file, mạng, CSDL — luôn chậm hơn nhiều so với tính toán trong bộ nhớ (RAM), nên cần biết khi nào thao tác I/O đang là điểm nghẽn.</p>
<h2>▸ Entry — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Nắm tổng quan kiến trúc hệ điều hành (tiến trình là gì, bộ nhớ được quản lý ra sao ở mức khái niệm), hiểu I/O là thao tác &quot;chậm&quot; cần cẩn thận.</p>
<p><strong>Ví dụ thực tế — nhận ra I/O là điểm chậm nhất trong một đoạn code.</strong></p>
<pre><code class="language-java"><span class="hljs-keyword">public</span> List&lt;String&gt; <span class="hljs-title function_">readLines</span><span class="hljs-params">(String filePath)</span> <span class="hljs-keyword">throws</span> IOException {
  List&lt;String&gt; lines = <span class="hljs-keyword">new</span> <span class="hljs-title class_">ArrayList</span>&lt;&gt;();
  <span class="hljs-keyword">try</span> (<span class="hljs-type">BufferedReader</span> <span class="hljs-variable">reader</span> <span class="hljs-operator">=</span> <span class="hljs-keyword">new</span> <span class="hljs-title class_">BufferedReader</span>(<span class="hljs-keyword">new</span> <span class="hljs-title class_">FileReader</span>(filePath))) {
    String line;
    <span class="hljs-keyword">while</span> ((line = reader.readLine()) != <span class="hljs-literal">null</span>) {
      lines.add(line);   <span class="hljs-comment">// xử lý trong RAM — rất nhanh</span>
    }
  }   <span class="hljs-comment">// đọc file (I/O) — chậm hơn xử lý trong RAM hàng nghìn lần</span>
  <span class="hljs-keyword">return</span> lines;
}
</code></pre>
<p>Bạn hiểu: nếu chương trình chạy chậm, thường không phải vì vòng lặp xử lý dữ liệu trong RAM chậm, mà vì thao tác I/O (đọc file, gọi API, truy vấn CSDL) đang chiếm phần lớn thời gian — nên khi tối ưu, ưu tiên nhìn vào I/O trước.</p>
<p><strong>Vì sao là mức ①:</strong> nắm khái niệm đủ để hiểu chương trình chạm vào tầng nào của hệ điều hành, chưa tự tối ưu hiệu năng ở tầng này.</p>
<h2>▸ Ex·V1 — ② Biết làm</h2>
<p><strong>Khác Entry:</strong> tự xử lý được I/O cơ bản trong code — đọc/ghi file, gọi API bên ngoài — đúng cách (đóng resource, xử lý lỗi).</p>
<p><strong>Ví dụ thực tế — dùng try-with-resources để tự đóng file, tránh rò rỉ tài nguyên.</strong></p>
<pre><code class="language-java"><span class="hljs-comment">// ❌ Quên đóng file nếu có lỗi giữa chừng → rò rỉ tài nguyên (file handle)</span>
<span class="hljs-type">FileReader</span> <span class="hljs-variable">reader</span> <span class="hljs-operator">=</span> <span class="hljs-keyword">new</span> <span class="hljs-title class_">FileReader</span>(path);
<span class="hljs-comment">// ... nếu lỗi ở đây, reader không bao giờ được đóng ...</span>
reader.close();

<span class="hljs-comment">// ✅ try-with-resources: TỰ ĐỘNG đóng dù có lỗi hay không</span>
<span class="hljs-keyword">try</span> (<span class="hljs-type">FileReader</span> <span class="hljs-variable">reader</span> <span class="hljs-operator">=</span> <span class="hljs-keyword">new</span> <span class="hljs-title class_">FileReader</span>(path)) {
  <span class="hljs-comment">// xử lý</span>
}   <span class="hljs-comment">// Java tự gọi reader.close() ở đây, kể cả khi có exception</span>
</code></pre>
<p>Bạn hiểu I/O (file, network) là tài nguyên hữu hạn cần đóng đúng cách — quên đóng nhiều lần sẽ làm hệ thống dần cạn tài nguyên (file handle, kết nối) dù mỗi lần chỉ rò một chút.</p>
<p><strong>Vì sao là mức ②:</strong> tự xử lý I/O đúng cách trong code thật — không chỉ hiểu khái niệm.</p>
<h2>▸ Specialist·V1 — ③ Thành thạo</h2>
<p><strong>Khác Ex·V1:</strong> hiểu rõ cơ chế kiểm soát bộ nhớ (Garbage Collector) và dùng công cụ phân tích heap để chẩn đoán rò rỉ bộ nhớ.</p>
<p><strong>Ví dụ thực tế — dùng heap dump để tìm rò rỉ bộ nhớ khiến service ngày càng chậm.</strong></p>
<pre><code class="language-bash">jmap -dump:live,format=b,file=heap.bin &lt;pid&gt;
</code></pre>
<p>Mở file bằng Eclipse MAT (Memory Analyzer Tool), thấy một <code>Map</code> tĩnh (<code>static Map&lt;String, Order&gt; cache</code>) giữ tham chiếu tới hàng triệu object <code>Order</code> cũ mà không bao giờ được dọn — vì code chỉ thêm vào cache mà quên đặt cơ chế hết hạn/xoá. Garbage Collector không thể dọn những object vẫn còn bị tham chiếu từ một biến <code>static</code> sống suốt vòng đời ứng dụng.</p>
<p><strong>Vì sao là mức ③:</strong> bạn chẩn đoán được vấn đề bộ nhớ bằng công cụ chuyên biệt — không chỉ xử lý I/O đúng cách ở mức code thông thường.</p>
<h2>▸ Specialist·V3 — ④ Chuyên sâu</h2>
<p><strong>Khác V1:</strong> tối ưu <strong>cấu hình tham số</strong> hệ điều hành/JVM/webserver ở tầm hệ thống — đúng nhiệm vụ &quot;tối ưu hệ thống (tham số cấu hình HĐH/DB/webserver/framework)&quot; mà career-path Specialist nhấn mạnh.</p>
<p><strong>Ví dụ thực tế — điều chỉnh kích thước heap JVM dựa trên đo đạc thật, không theo mặc định.</strong></p>
<pre><code class="language-bash"><span class="hljs-comment"># Mặc định JVM tự chọn heap size dựa trên RAM máy — không phải lúc nào cũng tối ưu</span>
java -Xms2g -Xmx4g -jar app.jar   <span class="hljs-comment"># đặt heap tối thiểu/tối đa dựa trên đo GC log thật</span>
</code></pre>
<p>Bạn đo GC log thấy Garbage Collector chạy quá thường xuyên (heap quá nhỏ so với lượng object service tạo ra) — điều chỉnh <code>-Xmx</code> dựa trên số liệu đo, không đoán một con số &quot;an toàn&quot; chung chung. Đặt quá lớn cũng có hại (GC pause lâu hơn khi dọn heap lớn).</p>
<p><strong>Vì sao là mức ④:</strong> bạn tối ưu được cấu hình hệ thống dựa trên đo đạc sâu — mức cao nhất của kỹ năng này trong toàn thang BE.</p>
