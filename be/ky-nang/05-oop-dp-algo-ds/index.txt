18:T24f6,<h1>OOP / Design Pattern / Algorithm / Data Structure</h1>
<p><strong>Định nghĩa.</strong> Bốn nền tảng khoa học máy tính dùng chung cho mọi dev:</p>
<ul>
<li><strong>OOP</strong>: tổ chức code quanh <em>đối tượng</em> với đóng gói (encapsulation), kế thừa (inheritance), đa hình.</li>
<li><strong>Design Pattern</strong>: lời giải mẫu cho bài toán thiết kế lặp lại (Creational: Singleton, Factory; Structural: Adapter).</li>
<li><strong>Algorithm</strong>: cách giải + đánh giá chi phí (độ phức tạp — Big-O).</li>
<li><strong>Data Structure</strong>: cách tổ chức dữ liệu (kiểu cơ bản, mảng...) — chọn đúng thì nhanh &amp; gọn.</li>
</ul>
<p>Ở Back-end (Java) chúng không hàn lâm: chọn đúng cấu trúc và pattern quyết định service chạy nhanh, dễ bảo trì hay không.</p>
<h2>▸ Entry — ① Nhập môn</h2>
<p><strong>Ở mức này bạn làm chủ được gì.</strong> Hiểu class/method/encapsulation/inheritance; pattern đơn giản (Singleton, Factory, Adapter); đánh giá độ phức tạp cơ bản (O(n) vs O(n²)); cài đặt được thuật toán sắp xếp/tìm kiếm cơ bản.</p>
<p><strong>Ví dụ thực tế — Singleton cho một service dùng chung (Spring tự làm điều này).</strong></p>
<pre><code class="language-java"><span class="hljs-meta">@Service</span>   <span class="hljs-comment">// Spring mặc định tạo DUY NHẤT 1 instance của service này (singleton) trong toàn ứng dụng</span>
<span class="hljs-keyword">public</span> <span class="hljs-keyword">class</span> <span class="hljs-title class_">EmailService</span> {
  <span class="hljs-keyword">public</span> <span class="hljs-keyword">void</span> <span class="hljs-title function_">send</span><span class="hljs-params">(String to, String subject)</span> { <span class="hljs-comment">/* ... */</span> }
}
</code></pre>
<p>Bạn không cần tự viết pattern Singleton thủ công (private constructor + static instance) — Spring's <code>@Service</code>/<code>@Component</code> đã áp dụng sẵn nguyên lý này, hiểu bản chất giúp bạn biết vì sao inject cùng 1 service ở nhiều nơi vẫn dùng chung 1 instance.</p>
<p><strong>Ví dụ 2 — độ phức tạp: tránh vòng lặp lồng nhau không cần thiết.</strong></p>
<pre><code class="language-java"><span class="hljs-comment">// ❌ O(n*m): với mỗi order lại duyệt lại toàn bộ user</span>
<span class="hljs-keyword">for</span> (Order o : orders) {
  <span class="hljs-keyword">for</span> (User u : users) { <span class="hljs-keyword">if</span> (u.getId().equals(o.getUserId())) o.setUserName(u.getName()); }
}
<span class="hljs-comment">// ✅ O(n+m): dựng Map tra cứu O(1)</span>
Map&lt;Long, String&gt; nameById = users.stream().collect(Collectors.toMap(User::getId, User::getName));
orders.forEach(o -&gt; o.setUserName(nameById.get(o.getUserId())));
</code></pre>
<p><strong>Vì sao là mức ①:</strong> nắm khái niệm, chọn đúng cấu trúc/pattern cơ bản cho bài toán đơn giản.</p>
<h2>▸ Ex·V1 — ② Biết làm</h2>
<p><strong>Khác Entry:</strong> tự thiết kế class hướng đối tượng, implement được pattern cơ bản (không chỉ nhận diện), dùng đúng cấu trúc dữ liệu (Array, ArrayList, Map) vào task thực tế.</p>
<p><strong>Ví dụ thực tế — implement Factory pattern cho việc tạo thông báo theo kênh khác nhau.</strong></p>
<pre><code class="language-java"><span class="hljs-keyword">interface</span> <span class="hljs-title class_">Notifier</span> { <span class="hljs-keyword">void</span> <span class="hljs-title function_">send</span><span class="hljs-params">(String to, String message)</span>; }

<span class="hljs-keyword">class</span> <span class="hljs-title class_">EmailNotifier</span> <span class="hljs-keyword">implements</span> <span class="hljs-title class_">Notifier</span> {
  <span class="hljs-keyword">public</span> <span class="hljs-keyword">void</span> <span class="hljs-title function_">send</span><span class="hljs-params">(String to, String message)</span> { <span class="hljs-comment">/* gửi email */</span> }
}
<span class="hljs-keyword">class</span> <span class="hljs-title class_">SmsNotifier</span> <span class="hljs-keyword">implements</span> <span class="hljs-title class_">Notifier</span> { <span class="hljs-comment">/* ... */</span> }

<span class="hljs-keyword">class</span> <span class="hljs-title class_">NotifierFactory</span> {
  <span class="hljs-keyword">static</span> Notifier <span class="hljs-title function_">create</span><span class="hljs-params">(String channel)</span> {
    <span class="hljs-keyword">return</span> <span class="hljs-keyword">switch</span> (channel) {
      <span class="hljs-keyword">case</span> <span class="hljs-string">&quot;EMAIL&quot;</span> -&gt; <span class="hljs-keyword">new</span> <span class="hljs-title class_">EmailNotifier</span>();
      <span class="hljs-keyword">case</span> <span class="hljs-string">&quot;SMS&quot;</span> -&gt; <span class="hljs-keyword">new</span> <span class="hljs-title class_">SmsNotifier</span>();
      <span class="hljs-keyword">default</span> -&gt; <span class="hljs-keyword">throw</span> <span class="hljs-keyword">new</span> <span class="hljs-title class_">IllegalArgumentException</span>(<span class="hljs-string">&quot;Kênh không hỗ trợ: &quot;</span> + channel);
    };
  }
}
</code></pre>
<p>Bạn tự implement pattern (Factory) để thêm kênh mới (vd Zalo) chỉ cần thêm 1 class + 1 case, không sửa code gọi.</p>
<p><strong>Vì sao là mức ②:</strong> tự thiết kế và implement được pattern cơ bản cho bài toán thực tế, không chỉ nhận diện pattern có sẵn.</p>
<h2>▸ Ex·V3 — ③ Thành thạo</h2>
<p><strong>Khác V1:</strong> hiểu chuyên sâu hơn — implement được pattern nâng cao hơn (Builder, Prototype, Bridge, Decorator, Facade, Proxy), nhận biết giải thuật cây/đồ thị/chia-để-trị, đánh giá space-time tradeoff của cấu trúc dữ liệu.</p>
<p><strong>Ví dụ thực tế — Builder pattern cho object có nhiều tham số tuỳ chọn.</strong></p>
<pre><code class="language-java"><span class="hljs-type">Order</span> <span class="hljs-variable">order</span> <span class="hljs-operator">=</span> Order.builder()
    .customerId(<span class="hljs-number">42L</span>)
    .items(items)
    .discountCode(<span class="hljs-string">&quot;SALE10&quot;</span>)   <span class="hljs-comment">// tham số tuỳ chọn — có thể bỏ qua</span>
    .build();
</code></pre>
<p>Thay vì constructor với 10 tham số (dễ truyền nhầm thứ tự, khó đọc), Builder cho phép set từng phần rõ ràng theo tên, bỏ qua tham số không cần.</p>
<p><strong>Ví dụ 2 — nhận biết bài toán cần giải thuật chia-để-trị.</strong> Tìm 2 sản phẩm trong danh sách đã sắp xếp có tổng giá đúng bằng ngân sách khách đưa ra — bạn nhận ra đây là bài toán &quot;two pointers&quot; (O(n)) thay vì so từng cặp (O(n²)).</p>
<p><strong>Vì sao là mức ③:</strong> bạn vận dụng được pattern và giải thuật ở mức nâng cao hơn, đánh giá đúng đánh đổi — không chỉ implement pattern cơ bản.</p>
<h2>▸ Senior·V3 — ④ Chuyên sâu</h2>
<p><strong>Khác Ex·V3:</strong> áp pattern ở <strong>tầm kiến trúc</strong> (nhiều module/nhà cung cấp), không chỉ trong phạm vi một class.</p>
<p><strong>Ví dụ thực tế — Adapter pattern gộp 3 cổng thanh toán khác nhau sau một hợp đồng chung.</strong></p>
<pre><code class="language-java"><span class="hljs-keyword">interface</span> <span class="hljs-title class_">PaymentGateway</span> { PaymentResult <span class="hljs-title function_">charge</span><span class="hljs-params">(<span class="hljs-type">long</span> amount, String orderId)</span>; }

    <span class="hljs-keyword">return</span> <span class="hljs-keyword">new</span> <span class="hljs-title class_">PaymentResult</span>(res.getResultCode() == <span class="hljs-number">0</span>, res.getTransId());   <span class="hljs-comment">// dịch response riêng của Momo</span>
  }

</code></pre>
<p>Thêm cổng thanh toán thứ 4 chỉ cần viết thêm một Adapter, không sửa code checkout đang chạy.</p>
<p><strong>Vì sao là mức ④:</strong> bạn dùng design pattern giải quyết bài toán kiến trúc thật (nhiều nhà cung cấp, dễ mở rộng) — không chỉ áp dụng pattern trong phạm vi một class.</p>
